> **涉及文件**：`combat_manager.gd`、`sprite_animator.gd`、`unit_base.gd`

## 问题一：两个单位重叠到同一格

### 根因分析

经追踪代码，发现**三条路径**可导致格子重叠：

#### 路径 A：`_get_unit_cell()` 的隐式重占（行 1073-1097）

当单位的 `_unit_cell` 记录丢失或过期时，`_get_unit_cell()` 会用 `world_to_axial(unit.position)` 重算格坐标，然后调用 `_occupy_cell(resolved_cell, unit)` 隐式注册。

**问题**：`play_quick_cell_step()` 使用 Tween 平滑移动单位的 `position`，在 Tween 进行过程中，`unit.position` 处于**旧格和新格之间的中间位置**。如果此时 `_unit_cell` 记录因某种原因丢失（例如被 `_pre_tick_scan` 的 `_trim_runtime_caches` 清理），`_get_unit_cell()` 的 `world_to_axial` 会返回一个**中间格**，而这个格可能已被另一个单位占据。`_find_nearest_free_cell` 可能选到旁边的格，导致逻辑上的格子记录与视觉 position 不一致，最终多个单位在视觉上重叠。

```gdscript
# 行 1084：单位在 Tween 中途，position 处于中间值
var fallback_cell: Vector2i = _hex_grid.call("world_to_axial", unit.position)
# fallback_cell 可能返回中间格，而非逻辑上已占的目标格
```

#### 路径 B：`_register_units()` 的初始放置（行 540-569）

`_register_units()` 中对每个单位调用 `_find_nearest_free_cell(cell)` 寻找空格。但 BFS 搜索基于 `_is_cell_free()`，而 `_is_cell_free()` 查 `_cell_occupancy`。如果多个单位初始 position 映射到**同一个格坐标**，第一个占成功后第二个 BFS 找到旁边空格——这本身没问题。但如果它们的 **world position 完全相同**，视觉上看就是重叠的（即使逻辑格不同，视觉位置没吸附到格心）。

检查行 566-568：
```gdscript
if resolved_cell.x >= 0 and _occupy_cell(resolved_cell, unit):
    var unit_node: Node2D = unit as Node2D
    if unit_node != null:
        unit_node.position = _hex_grid.call("axial_to_world", resolved_cell)
```
这里**会**把 position 吸附到格心，所以这条路径理论上不会导致视觉重叠——但存在一个隐患：如果 `unit` 不是 `Node2D`（`as Node2D` 返回 null），position 就不会被更新。

#### 路径 C：`_occupy_cell()` 自身的旧格清理漏洞（行 1029-1034）

```gdscript
if _unit_cell.has(iid):
    var old_cell: Vector2i = _unit_cell[iid]
    if old_cell != cell:
        var old_key: int = _cell_key_int(old_cell)
        if _cell_occupancy.has(old_key) and int(_cell_occupancy[old_key]) == iid:
            _cell_occupancy.erase(old_key)
```

条件 `int(_cell_occupancy[old_key]) == iid` 保证只清自己。但如果旧格已经被别人覆盖占了（`_cell_occupancy[old_key]` != `iid`），旧格不会被清理——**这本身是正确的**。但问题是 `_unit_cell[iid]` 仍然指向 `old_cell`，虽然即将被覆盖为 `cell`。

真正的漏洞场景：
1. 单位 A 逻辑占格 X，Tween 正在移位
2. `_pre_tick_scan` → 增量遍历 → `_cache_alive_unit_incremental` → `_spatial_hash.update(iid, unit.position)` 用的是实时 position（中间位）
3. 其他单位的 `_pick_target_in_attack_range` 通过 `_spatial_hash.query_radius` 查到 A 在中间位置
4. 不影响占格——但 `_get_unit_cell(A)` 如果被调到，会发现 `_unit_cell[A]` = X 且 `is_inside_grid(X)` = true → 返回 X → 正确
5. **所以路径 C 本身不是重叠的根因**

### 真正的重叠根因

经以上分析，**核心根因是路径 A**：`_get_unit_cell()` 的隐式重占。但触发条件是 `_unit_cell` 记录丢失。让我们追踪何时会丢失：

- `_vacate_cell(cell)` — 行 1046-1049：如果格子里的占用者不是自己，会 erase `_unit_cell`
- `_vacate_unit(unit)` — 行 1052-1062：死亡时调用
- `_remove_runtime_entry_by_id(iid)` — 死亡/清理时调用
- `_trim_runtime_caches` — 清理无效单位

**但还有一个场景被遗漏**：即使 `_unit_cell` 没丢失，如果两个单位的 Tween 完成时机恰好使它们的 `position` 都落在同一个格心区域，而其中一个的 `_unit_cell` 记录已被移动到新格但 position 还没跟上（Tween 被 kill 或被覆盖），就会出现**视觉重叠但逻辑格不同**的情况。

从截图来看，这正是问题：棋子视觉上在同一格，但逻辑上可能占了不同格。

### 修改建议

**修复 1（核心）：战斗结束后强制吸附到逻辑格心**

在 `_cleanup_unit_after_battle()` 中，对每个存活单位强制把 position 吸附到其逻辑格的格心：

```gdscript
# combat_manager.gd — _cleanup_unit_after_battle() 修改：
func _cleanup_unit_after_battle(unit: Node, winner_team: int) -> void:
    var movement: Node = _get_movement(unit)
    if movement != null:
        movement.call("clear_target")

    if unit.has_method("kill_quick_step_tween"):
        unit.call("kill_quick_step_tween")
    var iid: int = unit.get_instance_id()
    if _hex_grid != null and _unit_cell.has(iid):
        var cell: Vector2i = _unit_cell[iid]
        var snap_pos: Vector2 = _hex_grid.call("axial_to_world", cell)
        var unit_node: Node2D = unit as Node2D
        if unit_node != null:
            unit_node.position = snap_pos

    # ... 后续 IDLE/VICTORY 动画切换 ...
```

```gdscript
# unit_base.gd — 新增方法：
func kill_quick_step_tween() -> void:
    if _quick_step_tween != null and _quick_step_tween.is_valid():
        _quick_step_tween.kill()
    _quick_step_tween = null
```

**修复 2（防御性）：`_get_unit_cell()` 禁止隐式重占，仅做只读查询**

当前 `_get_unit_cell()` 既是查询又是副作用操作（会 `_occupy_cell` + 修改 position）。拆分为读写两个函数：

```gdscript

# 带注册副作用的版本（仅在 _register_units / 初始化时使用）：
func _resolve_and_register_unit_cell(unit: Node) -> Vector2i:
    # ... 原来的 _get_unit_cell 的 fallback 逻辑 ...
```

**修复 3（每逻辑帧校验）：在 `_pre_tick_scan` 中检测并修正重叠**

```gdscript
# _pre_tick_scan_incremental() 末尾新增：
_validate_cell_occupancy()

func _validate_cell_occupancy() -> void:
    # 正向检查：每个 _cell_occupancy 的 value(iid) 是否对应 _unit_cell[iid] 一致
    var cells_to_fix: Array[int] = []
    for raw_key in _cell_occupancy.keys():
        var cell_key: int = int(raw_key)
        var iid: int = int(_cell_occupancy[cell_key])
        if not _unit_cell.has(iid):
            cells_to_fix.append(cell_key)
            continue
        var expected_key: int = _cell_key_int(_unit_cell[iid])
        if expected_key != cell_key:
            cells_to_fix.append(cell_key)
    for key in cells_to_fix:
        _cell_occupancy.erase(key)
```

## 问题二：战斗结束后棋子倾斜/左右摇晃而非待机浮动

### 根因分析

调用链：`stop_battle` → `_cleanup_unit_after_battle` → `unit.call("play_anim_state", 0/6, {})` → `unit_base.play_anim_state()` → `sprite_animator.play_state()`

`sprite_animator.play_state()` 内部（行 97-134）：
```
1. _stop_tween()                    ← 停掉旧 Tween ✓
2. _clear_impulse_state()           ← 清脉冲 ✓
3. _normalize_base_before_state()   ← 恢复到 _rest_position
4. _cache_base_transform()          ← 把 _target.position 记为 _base_position ⬅ 关键
5. _state = state
6. _apply_loop_pose(0.0)            ← 开始新的循环动画
```

步骤 3 中 `_normalize_base_before_state` 恢复到 `_rest_position`——但 `_rest_position` **从未被更新**。虽然 `unit_base._sync_sprite_animator_rest_anchor()` 存在并尝试调用 `sync_rest_transform_to_current` 或 `update_rest_position`，但 **`sprite_animator.gd` 中这两个方法都不存在！** 因此 `_rest_position` 始终是单位首次创建或从对象池取出时的初始值。

步骤 4 的 `_cache_base_transform()`（行 250-264）：
```gdscript
_base_position = _target.position     # 从 visual_root 读
_base_scale = _target.scale
_base_rotation = _target.rotation     # ⬅ 如果此时 rotation 非零（MOVE 残留），
                                      #    IDLE 循环就会以倾斜角为基准！
```

但在步骤 3 中已经恢复了 `_rest_position`，所以步骤 4 读到的应该是正确的 rest 值——**除非 `_rest_position` 本身就是错的**。

**真正的原因**：`_rest_position` 在 `_cache_base_transform` 中的首次记录（行 259-264）发生在 `_ready()` 时。此时 `_target`（即 `visual_root`）的 position 和 rotation 是初始值 `(0, 0)` 和 `0.0`。

但如果单位经历了 MOVE 动画，MOVE 会修改 `_target.rotation`（行 192）。**当战斗结束时**，`_normalize_base_before_state` 将 rotation 恢复到 `_rest_rotation`（= 0.0），`_cache_base_transform` 记录 `_base_rotation = 0.0` → IDLE 以 rotation=0 为基准 → **这应该是正确的。**

**那为什么还会倾斜？**

答案在 `_cleanup_unit_after_battle` 中调用 `leave_combat()`（行 1305），`leave_combat()` 里**又调用了一次 `play_anim_state(0, {})`**（行 272）。但在此之前（行 1301/1303），**已经调用过一次** `play_anim_state(0/6, {})`。

第一次调用 `play_state(IDLE)` 会被正确处理。但随后 `leave_combat()` 第二次调用 `play_state(IDLE)` 时，由于 `_state == AnimState.IDLE` 且 `_is_loop_state(IDLE) == true` → **行 103 直接 return 了，不执行任何处理**。

这本身没问题。**但如果第一次调用的是 VICTORY（state=6）**（胜方），`leave_combat()` 又调用 `play_state(IDLE)`——此时 `_state == VICTORY` ≠ `IDLE`，会重新执行 `_normalize_base_before_state` + `_cache_base_transform`。

**关键 Bug 发生在以下时序**：

```
_cleanup_unit_after_battle:
  1. play_anim_state(6)  → VICTORY → _normalize + _cache → 开始 VICTORY 循环
     → VICTORY 动画修改 _target.scale 和 _target.position（行 203-213）
  2. leave_combat()
     → play_anim_state(0) → IDLE
       → _stop_tween()            ← 无 Tween，无效
       → _normalize_base_before_state(IDLE):
           _target.position = _rest_position  ← 恢复
           _target.rotation = _rest_rotation  ← 恢复
           _target.scale = _rest_scale        ← 恢复
       → _cache_base_transform():
           _base_position = _target.position  ← 应该正确了
```

但问题是 **步骤 1 和 2 在同一帧内执行**。VICTORY 动画的 `_process` 还没来得及运行（`_apply_loop_pose` 在下一帧才执行），但 `_cache_base_transform` 在步骤 1 中已经把 VICTORY 的初始姿态记录了。然后步骤 2 读到的 `_rest_position/scale/rotation` 是正确的……

**Wait — 实际上问题更简单**。让我重新检查 `_normalize_base_before_state`（行 219-236）：

```gdscript
if _has_rest_transform:
    _target.position = _rest_position
    _target.scale = _rest_scale
    _target.rotation = _rest_rotation
```

如果 `_rest_position` 是对的（`(0, 0)`），那 `_target.position` 会被设为 `(0, 0)`。但 **`_target` 是 `visual_root`（子节点），不是单位本身**！`visual_root.position` 应该是 `(0, 0)`（相对于父节点），这是对的。

**所以…真正的 bug 可能在于 `_has_rest_transform` 在某些情况下被设为 false，导致 `_normalize` 不执行恢复**。或者 `_rest_rotation` 被错误地记录为非零值。

让我追踪 `_has_rest_transform = true` 在何时设定：只在 `_cache_base_transform` 中，且只设一次（行 259-264）。之后不再修改。所以 `_rest_rotation` = `_target.rotation` at `_ready()` time = 0.0 → 应该正确。

**最终结论**：问题出在 **`_cache_base_transform()` 被调用时 `_target` 的状态不干净**。具体来说，如果 `_ready()` 时播放了 BENCH 动画（行 70 `play_state(AnimState.IDLE)`）且 BENCH 动画修改了 rotation（见 `set_on_bench_state` 行 245 播放 BENCH），然后 `_cache_base_transform` 在某个时机记录了 BENCH 的倾斜 rotation 作为 `_rest_rotation`。

**但最直接的修复方案是**：不管根因多复杂，**在战斗结束时强制清零 `visual_root` 的所有变换，然后重新播放 IDLE**。

### 修改建议

**修复方案（彻底）：`_cleanup_unit_after_battle` 中强制重置 visual_root transform**

```gdscript
# combat_manager.gd — _cleanup_unit_after_battle() 完整重写：
func _cleanup_unit_after_battle(unit: Node, winner_team: int) -> void:
    # 1. 停止所有移动
    var movement: Node = _get_movement(unit)
    if movement != null:
        movement.call("clear_target")

    # 2. 杀掉残留的格子移动 Tween
    if unit.has_method("kill_quick_step_tween"):
        unit.call("kill_quick_step_tween")

    # 3. 强制吸附 position 到逻辑格心
    var iid: int = unit.get_instance_id()
    if _hex_grid != null and _unit_cell.has(iid):
        var cell: Vector2i = _unit_cell[iid]
        var snap_pos: Vector2 = _hex_grid.call("axial_to_world", cell)
        var unit_node: Node2D = unit as Node2D
        if unit_node != null:
            unit_node.position = snap_pos

    # 4. 强制重置 visual_root 的 transform（清除所有动画残留）
    if unit.has_method("reset_visual_transform"):
        unit.call("reset_visual_transform")

    # 5. 播放正确的战后动画
    if _is_unit_alive(unit):
        var team_id: int = int(unit.get("team_id"))
        if winner_team != 0 and team_id == winner_team:
            unit.call("play_anim_state", 6, {}) # VICTORY
        else:
            unit.call("play_anim_state", 0, {}) # IDLE

    unit.set("is_in_combat", false)
    unit.call("leave_combat")
```

**`unit_base.gd` — 新增 `reset_visual_transform()`：**

```gdscript
func reset_visual_transform() -> void:
    # 彻底清除动画残留，恢复到化学基准态
    if visual_root != null:
        visual_root.position = Vector2.ZERO
        visual_root.scale = Vector2.ONE
        visual_root.rotation = 0.0
        if visual_root is CanvasItem:
            (visual_root as CanvasItem).modulate = Color(1, 1, 1, 1)

    # 强制重置 SpriteAnimator 的 rest 基准
    if sprite_animator != null:
        sprite_animator.call("force_reset_rest_transform")
```

**`sprite_animator.gd` — 新增 `force_reset_rest_transform()`：**

```gdscript
func force_reset_rest_transform() -> void:
    _resolve_target()
    if _target == null:
        return
    _rest_position = _target.position
    _rest_scale = _target.scale
    _rest_rotation = _target.rotation
    if _target is CanvasItem:
        _rest_modulate = (_target as CanvasItem).modulate
    _has_rest_transform = true
    _base_position = _rest_position
    _base_scale = _rest_scale
    _base_rotation = _rest_rotation
    _base_modulate = _rest_modulate
```

这个方法同时也是 `_sync_sprite_animator_rest_anchor()` 应该调用的方法——修复了之前 `sync_rest_transform_to_current` 和 `update_rest_position` 方法不存在的问题。

**`unit_base.gd` — 修复 `_sync_sprite_animator_rest_anchor()`：**

```gdscript
func _sync_sprite_animator_rest_anchor() -> void:
    if sprite_animator == null:
        return
    # 修复：使用实际存在的方法名
    if sprite_animator.has_method("force_reset_rest_transform"):
        sprite_animator.call("force_reset_rest_transform")
```

**`unit_base.gd` — `leave_combat()` 中移除冗余的 `play_anim_state` 调用：**

```gdscript
func leave_combat() -> void:
```

## 验证方式

1. **重叠验证**：50v200 大规模战斗后，检查是否有任何两个存活棋子视觉上重叠
2. **动画验证**：
   - 我方胜利 → 胜方棋子全部跳跃（VICTORY），败方棋子浮动（IDLE）
   - 所有棋子 rotation = 0，无倾斜
   - 所有棋子不应左右摆动
3. **格心吸附**：战斗结束后，所有棋子精确位于六角格中心
