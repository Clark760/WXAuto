## 3. AutoLoad 模块说明

### 3.1 EventBus（全局事件总线）

**职责**
- 提供系统间解耦通信能力，避免模块硬引用。
- 统一声明核心事件（阶段变更、场景切换、数据重载、Mod 加载、对象池事件）。

**已定义信号**
- `phase_changed(previous_phase, next_phase)`
- `scene_change_requested(scene_path)`
- `scene_changed(previous_scene, next_scene)`
- `data_reloaded(is_full_reload, summary)`
- `mod_loaded(mod_id, mod_name, load_order)`
- `mod_load_completed(summary)`
- `object_pool_registered(pool_key)`
- `object_pool_acquired(pool_key, instance_id)`
- `object_pool_released(pool_key, instance_id)`

**后续建议**
- M1/M2 新系统接入时优先通过 EventBus 发布状态事件，不直接调用 UI/战斗层。

---

### 3.2 GameManager（游戏总管理）

**职责**
- 维护阶段状态机：`BOOT / MAIN_MENU / PREPARATION / COMBAT / RESULT / PAUSED`
- 启动流程：基础数据加载 -> Mod 加载 -> 进入主场景
- 提供统一切场景入口

**核心接口**
- `initialize_game()`
- `reload_game_data()`
- `set_phase(next_phase)`
- `request_scene_change(scene_path)`
- `change_scene(scene_path)`
- `get_phase_name(phase)`

**实现细节**
- 初始切场景使用 `call_deferred`，避免 `_ready()` 时机下树结构忙碌错误。

**后续建议**
- M2 战斗阶段开启时，只通过 `set_phase()` 控制全局状态，不在业务层自行写枚举状态。

---

### 3.3 ObjectPool（通用对象池）

**职责**
- 提供对象复用能力，降低高频创建销毁开销。
- 适用于单位对象、VFX、伤害数字、UI 临时节点。

**核心接口**
- `register_factory(pool_key, factory, prewarm_count, default_parent)`
- `acquire(pool_key, parent_override)`
- `release(pool_key, node)`
- `clear_pool(pool_key, free_nodes)`
- `get_pool_stats(pool_key)`
- `get_all_pool_stats()`

**后续建议**
- M1 `UnitFactory` 与 M2 `VfxFactory` 直接以 `pool_key` 区分池实例。
- 统一约定 `pool_key` 命名：`unit:<id>`、`vfx:<id>`、`ui:<type>`。

---

### 3.4 DataManager（数据总管理）

**职责**
- 扫描并解析 `data/` 下 JSON。
- 标准化记录结构（核心主键为 `id`）。
- 以 `category + id` 合并，后加载覆盖先加载。

**支持分类**
- `units`
- `gongfa`
- `equipment`
- `linkages`
- `levels`
- `vfx`

**核心接口**
- `load_base_data()`
- `load_category_from_dir(category, dir_path, source_tag)`
- `get_record(category, record_id)`
- `get_all_records(category)`
- `get_summary()`
- `get_summary_text()`
- `get_source_of(category, record_id)`

**实现细节**
- 会跳过 `_schema` 子目录。
- 支持 3 类 JSON 结构：数组、单对象、带 `items` 包裹对象（并有映射结构兜底）。

**后续建议**
- M1 起新增数据字段时，优先扩展 Schema，再更新解析逻辑，保持 Mod 兼容。

---

### 3.5 ModLoader（Mod 扫描与覆盖）

**职责**
- 扫描 `user://mods` 与 `res://mods`
- 读取 `mod.json` 并按 `load_order` 排序
- 将 Mod `data/*` 合并到 DataManager（后加载覆盖先加载）
- 建立资源覆盖映射（`assets/*`）

**核心接口**
- `load_and_apply_mods()`
- `discover_mods()`
- `get_loaded_mods()`
- `get_asset_override_map()`
- `resolve_asset_path(asset_relative_path)`

**后续建议**
- M1 角色贴图加载时统一先走 `resolve_asset_path()`，确保支持 Mod 覆盖美术资源。

## 4. 战场基础模块说明

### 4.1 HexGrid

**文件**
- `scripts/board/hex_grid.gd`

**能力**
- 轴坐标与世界坐标互转（Pointy-Top）
- `_draw()` 六边形网格可视化
- 网格边界校验

**核心接口**
- `axial_to_world(axial)`
- `world_to_axial(world_pos)`
- `is_inside_grid(axial)`
- `get_all_cells()`

**后续建议**
- M1 拖拽部署直接复用 `world_to_axial()` 做落点吸附。

---

### 4.2 SpatialHash

**文件**
- `scripts/board/spatial_hash.gd`

**能力**
- 按 `cell_size` 建桶
- 对象插入、更新、移除
- 半径查询与 AABB 查询

**核心接口**
- `insert(object_id, world_position)`
- `update(object_id, new_world_position)`
- `remove(object_id)`
- `query_radius(center, radius)`
- `query_aabb(rect)`

**后续建议**
- M2 寻敌逻辑先用 `query_radius` 做候选集，再做仇恨/距离/站位二次筛选。

## 8. 后续模块编码约束（建议执行）

1. 新增系统优先挂在 `scripts/` 对应目录，并通过 EventBus 接入，不直接相互调用。
2. 新增数据字段先改 Schema，再改解析，再补样例数据，最后更新 Mod 兼容说明。
3. 涉及高频创建对象必须优先接 ObjectPool，不直接 `new()+queue_free()`。
4. 场景切换统一走 `GameManager.request_scene_change()` 或 `change_scene()`。
5. 进入 M1 前，先执行一次 `tools/check.ps1` 作为基线，避免在坏状态上继续开发。
