# M5：关卡系统详细设计

> **目标读者**：AI 代码生成器。本文档定义关卡系统的完整机制、数据结构与代码框架。  
> **前置依赖**：M0–M4 全部完成。

---

## 一、系统概述

关卡系统负责驱动整局游戏的 **PVE 推进节奏**：定义每一关的棋盘大小、敌人阵容、胜利奖励、地形障碍以及 Boss 战特殊机制。它是串联"布阵期 → 交锋期 → 结算 → 下一关"完整流程的核心骨架。

### 设计原则

| 原则 | 说明 |
|------|------|
| **数据驱动** | 所有关卡配置为 JSON，Mod 可覆盖/新增 |
| **高内聚低耦合** | 关卡管理器只负责"读配置 → 通知各系统"，不包含战斗/经济/功法具体逻辑 |
| **渐进难度** | 普通→精英→Boss→休息，节奏参考云顶之弈 |
| **可扩展** | Boss 机制、地形类型、奖励类型均可通过 JSON 扩展 |

## 二、关卡 JSON 数据结构

### 2.1 关卡配置文件

路径：`data/stages/stage_{chapter}_{index}.json`（一章多关）或 `data/stages/stages.json`（单文件数组）

```json
{
  "id": "stage_1_3",
  "chapter": 1,
  "index": 3,
  "name": "竹林伏击",
  "type": "normal",
  "description": "唐门暗器师埋伏于竹林之中",

  "grid": {
    "width": 32,
    "height": 16,
    "deploy_zone": {
      "x_min": 0, "x_max": 15,
      "y_min": 0, "y_max": 15
    },
    "hex_size": 26.0
  },

  "enemies": [
    { "unit_id": "tangmen_archer", "count": 8, "star": 1, "deploy_zone": "back" },
    { "unit_id": "tangmen_assassin", "count": 4, "star": 1, "deploy_zone": "front" },
    { "unit_id": "tangmen_elder", "count": 1, "star": 2, "deploy_zone": "center",
      "gongfa_ids": ["gongfa_tangmen_anqi"], "equip_ids": ["equip_yufeng_needle"] }
  ],

  "obstacles": [
    { "type": "bamboo", "cells": [[10,5],[10,6],[11,5],[11,6],[12,5]] },
    { "type": "rock",   "cells": [[20,8],[20,9]] }
  ],

  "rewards": {
    "silver": 10,
    "exp": 2,
    "drops": [
      { "type": "gongfa",    "pool": ["gongfa_tangmen_01","gongfa_tangmen_02"], "count": 1, "chance": 0.4 },
      { "type": "equipment", "pool": ["equip_basic_sword","equip_basic_armor"], "count": 1, "chance": 0.6 },
      { "type": "unit",      "pool": ["tangmen_archer"], "count": 1, "chance": 0.2, "star": 1 }
    ]
  },

  "boss_mechanics": null,

  "round_income": {
    "base_silver": 5,
    "interest_cap": 5
  }
}
```

### 2.2 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 唯一标识 |
| `chapter` | int | 章节编号，用于 UI 分组 |
| `index` | int | 章内关卡序号 |
| `name` | string | 关卡显示名 |
| `type` | enum | `"normal"` / `"elite"` / `"boss"` / `"rest"` / `"event"` |
| `grid` | object | 棋盘配置（见 §三） |
| `enemies` | array | 敌人配置（见 §四） |
| `obstacles` | array | 中立障碍物（见 §六） |
| `rewards` | object | 奖励配置（见 §五） |
| `boss_mechanics` | object/null | Boss 特殊机制（见 §七） |
| `round_income` | object | 本关布阵期收入参数 |

### 2.3 关卡序列配置

路径：`data/stages/stage_sequence.json`

```json
{
  "chapters": [
    {
      "chapter": 1,
      "name": "初入江湖",
      "stages": ["stage_1_1","stage_1_2","stage_1_3","stage_1_4_elite","stage_1_5_boss"],
      "rest_after": true
    },
    {
      "chapter": 2,
      "name": "唐门之乱",
      "stages": ["stage_2_1","stage_2_2","stage_2_3","stage_2_4_elite","stage_2_5","stage_2_6_boss"]
    }
  ]
}
```

## 三、格子大小配置

### 3.1 机制

每个关卡通过 `grid` 字段独立配置棋盘参数：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `width` | 32 | 战场横向格数 |
| `height` | 16 | 战场纵向格数 |
| `deploy_zone` | 左半区 | 我方可部署区域的格子坐标范围 |
| `hex_size` | 26.0 | 六角格像素半径 |

### 3.2 实现

`StageManager` 在加载关卡时调用 `HexGrid.resize(width, height, hex_size)` 重新生成网格。`BattlefieldRuntime` 读取 `deploy_zone` 限制拖拽部署区域。

## 四、敌人配置

### 4.1 机制

`enemies` 数组中每个条目定义一类敌人：

| 参数 | 类型 | 说明 |
|------|------|------|
| `unit_id` | string | 角色 JSON ID |
| `count` | int | 该类敌人数量 |
| `star` | int | 星级（1-3），影响属性倍率 |
| `deploy_zone` | string | 部署区域：`"front"/"back"/"center"/"random"/"fixed"` |
| `fixed_cells` | array | 当 `deploy_zone="fixed"` 时指定精确格坐标 |
| `gongfa_ids` | array | 可选，覆盖该敌人的功法列表 |
| `equip_ids` | array | 可选，覆盖该敌人的装备列表 |
| `stat_scale` | float | 可选，属性缩放倍率（默认 1.0），用于精英/Boss 难度调节 |

### 4.2 部署区域语义

| 值 | 含义 |
|------|------|
| `"front"` | 敌方部署区前 1/3 行（近战先锋） |
| `"back"` | 敌方部署区后 1/3 行（远程输出） |
| `"center"` | 敌方部署区中心区域（Boss/统领） |
| `"random"` | 在敌方整个部署区随机分散 |
| `"fixed"` | 按 `fixed_cells` 精确放置 |

### 4.3 实现

`StageManager._spawn_stage_enemies()` 读取 `enemies` 数组，对每条记录调用 `UnitFactory.acquire_unit()` 创建敌人实例，根据 `deploy_zone` 调用 `_resolve_enemy_deploy_cells()` 计算实际格坐标。

## 五、奖励配置

### 5.1 机制

`rewards` 对象定义胜利后获得的奖励：

| 参数 | 类型 | 说明 |
|------|------|------|
| `silver` | int | 固定银两奖励 |
| `exp` | int | 固定经验奖励 |
| `drops` | array | 随机掉落池 |

每个 `drops` 条目：

| 参数 | 类型 | 说明 |
|------|------|------|
| `type` | enum | `"gongfa"` / `"equipment"` / `"unit"` |
| `pool` | array | 可选物品 ID 池，随机抽取 |
| `count` | int | 掉落数量 |
| `chance` | float | 掉落概率 (0.0–1.0) |
| `star` | int | 仅 `type="unit"` 时有效，掉落角色星级 |

### 5.2 奖励侠客的放置逻辑（TFT 机制）

当 `type="unit"` 的奖励触发时，按以下优先级放置：

```
1. 备战席(bench)是否有空位？
   ├─ 有空位 → 放入备战席第一个空槽
   └─ 无空位 →
       2. 我方棋盘部署区是否有空格？
          ├─ 有空格 → 随机选一个空格放置
          │   (该角色立即上场，但不占用 max_deploy 配额之外，
          │    玩家在下一个布阵期必须手动管理)
          └─ 无空格 → 奖励丢失（极端情况，实际不太可能发生）
```

> **参考**：这与英雄联盟·云顶之弈中"备战席满→角色出现在棋盘上"的行为一致。

### 5.3 实现

`RewardManager.apply_stage_rewards(rewards_config)` 处理全部奖励，내부调用：
- `EconomyManager.add_silver(amount)` — 银两
- `EconomyManager.add_exp(amount)` — 经验
- `_roll_drops(drops)` — 逐条检查 `chance`，命中后从 `pool` 随机抽取
- `_grant_gongfa(gongfa_id)` — 功法直接加入玩家仓库
- `_grant_equipment(equip_id)` — 装备直接加入玩家仓库
- `_grant_unit(unit_id, star)` — 调用放置逻辑（§5.2）

## 六、中立障碍物

### 6.1 机制

障碍物是**不可摧毁、不可穿越**的中立地形，占据指定六角格，影响战斗 AI 的寻路和技能范围。

| 参数 | 类型 | 说明 |
|------|------|------|
| `type` | string | 障碍物类型 ID，决定外观 |
| `cells` | array | 占据的格坐标列表 `[[x,y], ...]` |

### 6.2 预置障碍物类型

| 类型 ID | 视觉 | 特殊规则 |
|---------|------|---------|
| `"rock"` | 灰色岩石 | 完全阻挡移动与攻击 |
| `"bamboo"` | 竹林 | 阻挡移动，但远程攻击可穿过（伤害 -30%） |
| `"water"` | 浅水 | 可通过但移动速度 -50% |
| `"fire_pit"` | 火坑 | 不可通过，相邻格每秒受 `10` 点火焰伤害 |

### 6.3 实现

- `ObstacleManager` 管理所有障碍物实例
- 关卡加载时，`StageManager` 调用 `ObstacleManager.spawn_obstacles(obstacles_config)`
- `ObstacleManager` 在 `HexGrid` 中标记被占格为 `blocked`
- `FlowField.build()` 和 `CombatManager` 的寻路/AI 读取 blocked 格信息
- 视觉上：障碍物使用简单的 `ColorRect` / `Polygon2D` / `Sprite2D` 程序化绘制

## 七、Boss 战特殊机制

### 7.1 概述

每个 Boss 关卡通过 `boss_mechanics` 字段配置一组特殊规则，增加战斗策略深度。

### 7.2 Boss 机制库

以下为预置的 Boss 机制类型，通过 `boss_mechanics.type` 指定：

---

#### 机制一：「天罡护体」— 相位护盾

```json
{
  "type": "phase_shield",
  "config": {
    "phases": [
      { "hp_threshold": 0.7, "shield_hp": 500, "buff_on_break": "atk_up_20" },
      { "hp_threshold": 0.3, "shield_hp": 1000, "buff_on_break": "berserk" }
    ]
  }
}
```

**规则**：Boss 血量降至阈值时生成护盾。护盾存在期间 Boss 免疫所有伤害。打碎护盾后 Boss 进入虚弱状态（受伤 +30%，持续 5 秒），同时 Boss 获得 `buff_on_break` 增益。

---

#### 机制二：「呼朋引伴」— 小兵增援

```json
{
  "type": "summon_wave",
  "config": {
    "trigger": "periodic",
    "interval_seconds": 15,
    "max_waves": 3,
    "wave_units": [
      { "unit_id": "minion_swordsman", "count": 6, "star": 1 },
      { "unit_id": "minion_archer", "count": 4, "star": 1 }
    ]
  }
}
```

**规则**：Boss 每隔 `interval_seconds` 秒从场边召唤一波增援小兵。最多召唤 `max_waves` 波。增援小兵死亡不计入胜负判定，击杀 Boss 即胜利。

---

#### 机制三：「天地为炉」— 地面灾害区域

```json
{
  "type": "hazard_zones",
  "config": {
    "hazard_type": "fire",
    "damage_per_second": 25,
    "radius_cells": 3,
    "move_interval_seconds": 8,
    "count": 2,
    "warning_seconds": 2.0
  }
}
```

**规则**：战场上定期出现 `count` 个灾害区域（火焰/毒雾/冰封），持续对区域内所有单位造成伤害。每 `move_interval_seconds` 秒随机移动到新位置。移动前会显示 `warning_seconds` 秒的预警标记。敌我均受影响，鼓励玩家调整阵型。

---

#### 机制四：「分身幻术」— Boss 分身

```json
{
  "type": "clone_split",
  "config": {
    "hp_threshold": 0.5,
    "clone_count": 2,
    "clone_hp_ratio": 0.3,
    "clone_atk_ratio": 0.5,
    "real_boss_hidden": true,
    "reveal_on_clone_death": true
  }
}
```

**规则**：Boss 血量降至 50% 时分裂为 `clone_count` 个分身（外观完全一致）。分身拥有 Boss 属性的一定比例。如果 `real_boss_hidden=true`，真身会隐藏在分身中，玩家需要同时击杀所有分身才能找到真身。击杀分身后真身显形。

---

#### 机制五：「绝地反击」— 狂暴计时器

```json
{
  "type": "enrage_timer",
  "config": {
    "enrage_after_seconds": 60,
    "pre_enrage_warning_seconds": 10,
    "enrage_buffs": {
      "atk_percent": 1.0,
      "spd_percent": 0.5,
      "damage_reduce_percent": 0.3
    },
    "soft_enrage_stages": [
      { "at_seconds": 30, "atk_percent": 0.3 },
      { "at_seconds": 45, "atk_percent": 0.6 }
    ]
  }
}
```

**规则**：Boss 战有时间限制。经过 `enrage_after_seconds` 秒后 Boss 进入狂暴状态，属性大幅提升（几乎无法击杀）。中间有软狂暴阶段逐步增强。倒逼玩家在限时内输出足够伤害。

---

#### 机制六：「天罡阵法」— Boss 专属阵法

```json
{
  "type": "boss_formation",
  "config": {
    "formation_cells": [[15,7],[15,8],[16,7],[16,8]],
    "buff_in_formation": {
      "damage_reduce_percent": 0.5,
      "hp_regen_per_second": 20
    },
    "formation_visible": true,
    "destroyable": false
  }
}
```

**规则**：Boss 站在特定阵法区域内时获得强大增益（减伤 50%、每秒回血）。玩家需要通过带位移技能的角色（如轻功角色的击退效果）将 Boss 推出阵法区域，或绕后切入。

---

### 7.3 机制组合

一个 Boss 关卡可以同时配置**多个机制**：

```json
{
  "boss_mechanics": [
    { "type": "phase_shield", "config": { "phases": [...] } },
    { "type": "enrage_timer", "config": { "enrage_after_seconds": 90 } }
  ]
}
```

### 8.4 与现有系统的接口

| 现有系统 | 交互方式 | 说明 |
|----------|---------|------|
| `BattlefieldRuntime` | `StageManager` 通过信号通知加载关卡 → `BattlefieldRuntime` 调用 `HexGrid.resize()` | 棋盘大小动态调整 |
| `CombatManager` | `battle_ended` 信号 → `StageManager.on_battle_ended()` | 战斗结算触发奖励 |
| `EconomyManager` | `RewardManager` 调用 `add_silver()`/`add_exp()` | 奖励发放 |
| `UnitFactory` | `RewardManager._grant_unit()` 调用 `acquire_unit()` | 奖励侠客 |
| `GongfaManager` | `StageManager` 加载关卡后 `prepare_battle()` 自动处理功法 | 无需额外接入 |
| `HexGrid` | 新增 `resize()` 方法、`set_cell_blocked()` 方法 | 支持动态棋盘和障碍物 |
| `FlowField` | 读取 `ObstacleManager.get_all_blocked_cells()` | 寻路避障 |
| `EventBus` | 新增信号：`stage_loaded`、`stage_completed` 等 | 跨系统通知 |

### 8.5 数据流时序

```
用户按 F6 开始 →
  StageManager.start_stage(id) →
    StageData.parse(json) →
    emit stage_loaded →
    BattlefieldRuntime:
      HexGrid.resize(w, h, hex_size) →
      ObstacleManager.spawn_obstacles() →
      _spawn_stage_enemies() →
      经济 round_income 设置 →
    emit stage_prepare_started →

用户布阵完毕，按 F6 →
  _start_combat() (已有流程) →
    BossMechanicRunner.start(mechanics) →
    emit stage_combat_started →

战斗结束 →
  CombatManager.battle_ended 信号 →
    StageManager.on_battle_ended(winner) →
      if 胜利:
        RewardManager.apply_stage_rewards() →
        emit stage_completed →
        UI 显示奖励面板 →
        advance_to_next_stage() 或 all_stages_cleared()
      if 失败:
        emit stage_failed →
        UI 显示失败面板
```

## 十、验证标准

1. **完整流程**：可从第 1 关打到最终关，中间有休息关和 Boss 关
2. **棋盘动态**：不同关卡棋盘大小明显不同
3. **障碍物**：障碍物正确阻挡移动和影响 AI 寻路
4. **奖励发放**：银两/经验/功法/装备/侠客正确发放
5. **侠客溢出**：备战席满时奖励侠客正确出现在棋盘上
6. **Boss 机制**：至少 3 种机制可正常触发和运行
7. **数据驱动**：修改 JSON 后重载即生效
8. **性能**：Boss 机制不影响 60fps
