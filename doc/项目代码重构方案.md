# 项目代码重构方案

> 角色定位：本文档不是“优化建议清单”，而是当前项目的重构主设计文档。后续所有结构调整、拆分实施、阶段性验收，都应以本文档为统一依据，而不是零散口头约定。

> 重构策略：**渐进式绞杀**。保持现有项目持续可运行、可回归、可回退，在兼容旧接口和旧数据的前提下，逐步把系统迁移到新的稳定分层。

---

## 第一章：现状诊断

### 1.1 当前项目的结构性问题

当前项目的主要问题不是单个文件写得“丑”或“长”，而是**架构边界已经失控**。现状可以归纳为以下几类：

1. **战场主对象职责叠加**
   - 当前战场对象存在多层继承链：
     - `scripts/board/battlefield.gd`
     - `scripts/ui/battlefield_ui.gd`
     - `scripts/battle/battlefield_runtime.gd`
   - 这条继承链把以下职责叠在同一个对象上：
     - 战场运行时
     - 战场 UI
     - 输入路由
     - 商店 / 关卡 / 库存 / 回收编排
     - 战斗结束与阶段切换
   - 结果是：任何一个局内需求，都可能继续把代码堆进同一条继承链，进一步扩大耦合面。

2. **UI 只有一半在场景里，另一半仍在代码里现拼**
   - `scenes/battle/battlefield.tscn` 已经定义了顶栏、商店主面板、详情面板、背包面板、tooltip、备战席等大量基础结构。
   - 但脚本中仍然大量使用 `Label.new()`、`Button.new()`、`VBoxContainer.new()`、`HBoxContainer.new()` 等方式动态拼装：
     - 库存卡片
     - 商店卡片
     - 功法槽 / 装备槽行
     - 过滤按钮
     - tooltip 列表项
   - 这导致 UI 无法在编辑器中完整所见即所得，任何布局调整都必须回到脚本读代码。

3. **超大 God Object 普遍存在**
   - 当前主要高风险文件：

| 文件 | 行数 | 问题 |
|---|---:|---|
| `scripts/ui/battlefield_ui.gd` | 2369 | UI、输入、详情、背包、日志、tooltip 过载 |
| `scripts/board/battlefield.gd` | 2068 | 正式战斗编排、商店、关卡、回收、库存、战斗面板混杂 |
| `scripts/gongfa/gongfa_manager.gd` | 1997 | 系统级旧入口仍未收口，装配、触发、效果、监听、日志、数据访问、状态机混杂 |
| `scripts/combat/combat_manager.gd` | 1553 | 注册、移动、攻击、地形、缓存、事件中继、结算全耦合 |
| `scripts/battle/battlefield_runtime.gd` | 909 | 战斗上下文装配、autoload 注入、运行时桥接仍集中在旧战场链路 |

4. **运行时装配泛滥**
   - 代码大量在 `_ready()` 或 `_bootstrap_*()` 中执行：
     - `.new() as Node`
     - `add_child(...)`
   - 典型例子：
     - `battlefield_runtime.gd` 运行时创建 `RuntimeUnitDeployManager`、`RuntimeDragController`、`RuntimeBattlefieldRenderer`
     - `battlefield.gd` 运行时创建 `RuntimeShopController`、`RuntimeStageBridge`、`RuntimeInventoryController`、`RuntimeRecycleController`
   - 这种做法的直接后果：
     - 依赖关系不可视
     - 场景树不可读
     - 调试困难
     - 编辑器无法准确反映系统结构

5. **动态调用过多，静态边界过弱**
   - 代码广泛依赖：
     - `call(...)`
     - `_get_root_node("...")`
     - `get_node_or_null(...)`
   - 动态调用本身不是原罪，但在当前项目里已经被用作**替代模块接口设计**。
   - 结果是：
     - 类型关系模糊
     - 重构成本被隐藏
     - 调用链难以追踪
     - 局部改动容易引发级联回归

6. **目录分层与实际职责不匹配**
   - 原始规划希望有明确的 `scenes/ui/`、应用分层、组合优于继承。
   - 实际上：
     - `scripts/ui/` 中仍然夹杂大量业务逻辑
     - `scripts/board/` 和 `scripts/battle/` 的边界不清晰
     - `scripts/gongfa/` 既是数据入口、又是战斗监听器、又是效果总线
   - 目录看似分层，实际仍是横向耦合。

### 1.2 根因判断

从项目演化角度看，当前结构问题的根因不是“代码质量不好”，而是以下几项长期趋势叠加造成：

1. 以**功能迭代优先**替代了**边界维护优先**
2. 使用了“先塞进主对象，后面再拆”的模式，但“后面”长期没有真正发生
3. 把运行时动态装配当作模块化，实际上只是把耦合转移成了隐藏耦合
4. UI 已经开始场景化，但没有明确规定“禁止新增代码拼 UI”
5. 超大脚本虽已开始局部拆 controller，但拆分后的控制器仍依附在旧 God Object 下，未形成真正的新架构

### 1.3 结论

项目已经进入**必须进行底层结构重构**的阶段。继续在现有结构上迭代，会出现以下后果：

1. 每次新功能都进一步扩大 `battlefield` / `gongfa_manager` / `combat_manager` 的职责
2. UI 调整成本持续升高
3. 输入冲突、事件错连、生命周期混乱会越来越频繁
4. 测试成本会越来越高，但问题定位速度越来越慢
5. 后续任何真正的大功能都会先被结构阻力拖死

## 第二章：重构目标与原则

### 2.1 重构目标

本次重构的目标不是“清理代码风格”，而是建立一个可以长期承载项目迭代的稳定结构：

1. **场景负责声明结构**
   - 所有可编辑 UI 必须在 `tscn` 或子场景中定义。
   - 场景树必须反映真实运行结构。

2. **脚本负责绑定与协调**
   - 场景脚本只做节点绑定、状态投影、输入转发、调用应用层接口。
   - 不再承担跨系统业务编排。

3. **应用层负责用例编排**
   - 战斗启动、阶段推进、商店刷新、库存操作、回收出售等应由应用层协调。

4. **领域层负责业务规则**
   - `combat` 只处理战斗模拟。
   - `gongfa` 只处理功法和效果规则。
   - `stage`、`economy`、`inventory`、`shop` 保持独立边界。

5. **基础设施负责装配与兼容**
   - 单例访问、动态调用、兼容 facade 应收口到基础设施与过渡层。

### 2.2 设计原则

1. **组合优于继承**
   - 废弃 `battlefield -> battlefield_ui -> battlefield_runtime` 这类职责叠层。

2. **scene-first UI**
   - 可以在编辑器中看到的 UI，必须尽量在编辑器中定义。

3. **接口显式化**
   - 核心领域内部不允许继续以 `call()` 代替接口设计。

4. **渐进迁移**
   - 不进行一次性重写，不中断现有开发，不破坏当前数据和玩法。

5. **兼容优先**
   - 前两阶段不修改 JSON schema、不修改 Mod 组织、不改外部语义。

6. **可回归、可回退**
   - 每个阶段必须具备明确退出条件和回归测试覆盖。

## 第三章：目标架构蓝图

### 3.1 分层结构

目标架构采用五层：

```text
Scene Layer
  ├─ 场景声明
  ├─ 子场景组合
  └─ 节点依赖可视化

Application Layer
  ├─ 用例编排
  ├─ 阶段协调
  ├─ 输入路由
  └─ 跨模块流程控制

Domain Layer
  ├─ Combat
  ├─ Gongfa
  ├─ Stage
  ├─ Economy
  ├─ Inventory
  └─ Shop

UI Layer
  ├─ Presenter
  ├─ ViewBinder
  ├─ ViewModel Adapter
  └─ 可复用 UI 子场景

Infrastructure Layer
  ├─ Facade / Compatibility
  ├─ Service Locator / DI Entry
  ├─ EventBus Adapter
  └─ Data Access / Runtime Wiring
```

### 3.2 依赖方向

依赖方向必须固定为：

```text
Scene -> UI / App
UI -> App
App -> Domain
App -> Infra
Domain -> Domain
Infra -> Domain
```

禁止出现：

1. UI 直接访问 `CombatManager` / `GongfaManager`
2. Domain 直接操作具体 UI 节点
3. 场景层直接跨模块写业务规则
4. Domain 到处直接 `_get_root_node("...")`

### 3.3 `scripts/domain/` 放置判定

`domain` 不是“听起来像业务逻辑的目录”，而是**纯领域规则目录**。后续目录归位必须按下面标准执行，而不是按文件名感觉移动：

1. 满足以下全部条件的实现，才允许放入 `scripts/domain/`：
   - 不依赖具体场景树结构
   - 不直接 connect / emit Godot runtime signal
   - 不直接访问 `SceneTree`、AutoLoad、HexGrid、VFXFactory、`Control`
   - 不要求持有 `Node` 生命周期、`add_child()`、动画播放、世界坐标驱动
   - 输入输出可以收敛为显式参数和返回值，而不是把 facade 当动态属性袋
2. 仍直接承担运行时桥接职责的实现，必须继续留在对应 runtime 目录：
   - `scripts/combat/`
   - `scripts/unit_augment/`
   - `scripts/app/`
   - `scripts/infra/`
3. 禁止只把文件从 `scripts/combat/` 挪到 `scripts/domain/combat/`，但内部仍大量依赖：
   - `owner.get(...)`
   - `owner.call(...)`
   - manager 私有字段
   - 场景节点和运行时组件
4. 目录迁移的正确顺序必须是：
   - 先去掉 runtime 依赖
   - 再改成显式参数
   - 最后才迁入 `scripts/domain/`

### 3.4 Combat 子模块的 domain 归位规则

当前 `Combat` 拆分后，子模块要按两层理解：

1. runtime / adapter 层
   - `combat_runtime_service.gd`
   - `combat_unit_registry.gd`
   - `combat_movement_service.gd`
   - `combat_attack_service.gd`
   - `combat_terrain_service.gd`
   - `combat_event_bridge.gd`
   - 这些文件仍直接依赖 `CombatManager`、单位组件、信号、HexGrid、VFX 或运行时缓存，因此 Phase 1 内不应为了目录整洁强行迁入 `scripts/domain/`。
2. 候选 domain 层
   - `cell_occupancy.gd`
   - `combat_pathfinding.gd`
   - `combat_targeting.gd`
   - `combat_metrics.gd`
   - `flow_field.gd`
   - 这些模块未来可以进入 `scripts/domain/combat/`，但前提是先去掉把 `CombatManager` 当动态属性容器的用法。

### 3.5 战场子系统目标结构

战场主场景应改为组合：

```text
BattlefieldScene
  ├─ BattlefieldCoordinator
  ├─ BattlefieldWorldController
  ├─ BattlefieldHudPresenter
  ├─ BattlefieldSceneRefs
  ├─ CombatFacade
  ├─ GongfaFacade
  ├─ StageFacade
  ├─ ShopFacade
  └─ UI SubScenes
```

职责定义：

1. `BattlefieldScene`
   - 只负责场景装配、节点引用、初始化顺序

2. `BattlefieldCoordinator`
   - 负责局内用例编排
   - 例如：开始战斗、战斗结束、推进关卡、刷新商店、打开详情

3. `BattlefieldWorldController`
   - 只处理世界交互
   - 例如：拖拽部署、镜头缩放、右键平移、战场单位点击

4. `BattlefieldHudPresenter`
   - 只处理 HUD、日志、详情、背包、商店数据投影

5. `BattlefieldSceneRefs`
   - 统一维护场景节点引用
   - 让其他层通过显式依赖拿到节点，而不是四处写路径

### 3.6 `scripts/app/`、`scripts/infra/`、`scenes/ui/` 的硬边界

除了 `domain` 之外，另外三个冻结目录也必须明确边界，避免后续继续混写：

1. `scripts/app/`
   - 只承接用例编排、coordinator、presenter 装配、场景与领域之间的流程协调
   - 允许依赖 `domain` 与 `infra`
   - 不允许承载纯业务规则
   - 不允许直接声明 UI 节点树
2. `scripts/infra/`
   - 只承接单例访问收口、兼容 facade、service locator、外部依赖适配、资源/运行时装配入口
   - 不允许承载业务规则
   - 不允许反向吞掉应用层编排
3. `scenes/ui/`
   - 只承接可编辑、可预览、可复用的 UI 子场景
   - 所有生产可见 UI 的结构性变更，原则上都应先落到这里
   - 脚本只允许做绑定、刷新、事件转发和池化复用，不允许继续长出业务编排

当前阶段的审计结论：

1. `scripts/app/` 仍是冻结目录，Phase 1 不强行把 `battlefield` 相关旧壳提前搬入，真正迁移放在 Phase 2。
2. `scripts/infra/` 仍是冻结目录，Phase 1 不为了“目录好看”把未收口的旧胶水提前搬入，真正迁移放在 Phase 4。
3. `scenes/ui/` 目前仍只有占位说明，说明项目还没有进入 Phase 3 的 UI 全量 scene-first 迁移；这不是当前阻塞，但后续新 UI 不得继续绕开它。

## 第四章：重点子系统拆分方案

### 4.1 战场场景重构

> 说明：本节只保留战场重构的架构背景与目标方向，不再作为 Phase 2 的施工细则。
> Phase 2 的批次、接口拆分、测试与验收，统一以 [phase2_战场主场景组合编排落地方案](./phase2_战场主场景组合编排落地方案.md) 为准。

#### 当前问题

1. 继承链承载了运行时、UI、正式局内编排三层职责
2. controller 和 manager 在运行时动态创建，导致场景树不可视
3. 输入优先级散落在多个类里
4. 商店、库存、回收、日志、详情等局内 UI 直接挂在战场根脚本上

#### 目标方案

1. 取消 `battlefield.gd -> battlefield_ui.gd -> battlefield_runtime.gd` 继承链
2. 新战场根脚本只做装配
3. 将输入、UI、编排分为独立组合对象
4. 所有运行时控制器改为：
   - 场景内显式子节点，或
   - 由装配器统一注入，但必须在结构上可追踪

#### 迁移边界

第一阶段不直接推翻 `Battlefield` 玩法，只先引入新组合骨架和 facade，让旧流程还能走通。

### 4.2 战斗核心拆分

#### 当前问题

`combat_manager.gd` 当前同时负责：

1. 单位注册
2. 战斗主循环
3. 移动
4. 攻击
5. 地形
6. 缓存
7. 事件中继
8. 结算
9. 运行时 metrics

#### 目标拆分

拆为以下服务：

1. `CombatSimulation`
   - 逻辑帧推进
   - 战斗阶段驱动

2. `CombatUnitRegistry`
   - 战场单位注册
   - 存活单位查询
   - 队伍计数

3. `CombatMovementService`
   - cell 占用
   - 强制移动
   - 寻路移动
   - 移动失败/成功上报

4. `CombatAttackResolver`
   - 选敌
   - 射程判断
   - 攻击尝试
   - 伤害结算回调

5. `CombatTerrainService`
   - 临时地形
   - 静态地形
   - 地形标签
   - 阻挡与可视更新

6. `CombatEventRelay`
   - Godot signal 发射
   - 外部战斗事件兼容
   - battle summary 汇总

#### Combat 与 Domain 的边界补充

1. `CombatManager` 和 runtime service 负责：
   - 逻辑帧驱动
   - `Node` 生命周期
   - 组件信号连接
   - VFX、动画、世界坐标同步
2. `scripts/domain/combat/` 未来只允许承接：
   - 流场/路径评分
   - 选敌规则
   - 占格判定
   - metrics 聚合
   - 不需要直接访问场景树的战斗规则算法
3. 如果某个 `combat` 子模块还依赖 manager 私有字段或 `owner.call(...)`，它仍然只是“待纯化的 runtime 子模块”，不算真正进入 domain。

#### 兼容策略

保留 `CombatManager` 作为 facade，在第一阶段继续对上层暴露现有方法名，内部逐步转发到新服务。

### 4.3 UnitAugment 系统拆分

#### 当前问题

`gongfa_manager.gd` 当前同时承担：

1. 数据重载
2. 装备 / 功法配置
3. 被动效果应用
4. 自动触发轮询
5. 技能释放
6. 事件监听
7. tag linkage
8. buff 生命周期联动
9. 日志事件发射

#### 目标拆分

拆为以下服务：

1. `UnitAugmentRegistry`
   - 功法 / Buff / 装备数据重载
   - 数据索引与安全副本查询

2. `UnitAugmentBuffManager`
   - Buff 生命周期
   - 源绑定光环
   - tick 请求生成

3. `UnitAugmentUnitStateService`
   - baseline stats 构建
   - 功法 / 装备 / Buff 来源聚合
   - runtime 状态回写

4. `UnitAugmentBattleRuntime`
   - battle lifecycle
   - `_process` 驱动
   - `bind_combat_context(...)` / `prepare_battle(...)`

5. `UnitAugmentTriggerRuntime`
   - 自动触发轮询
   - 事件触发派发
   - trigger 上下文装配

6. `UnitAugmentTagRegistryService`
   - tag -> index 注册表
   - reload 后版本同步

7. `UnitAugmentTagLinkageResolver` / `UnitAugmentTagLinkageScheduler`
   - tag linkage provider 收集
   - watcher、dirty 标记、节流调度

#### 兼容策略

系统级生产入口统一切到 `UnitAugmentManager`，不保留 `GongfaManager` alias。条目级 `gongfa` 接口名保留，用于维持数据外观和玩法语义稳定。

### 4.4 效果系统拆分

#### 当前问题

旧 `effect_engine.gd` 曾经把以下职责揉在一起：

1. 被动效果处理
2. 主动效果分发
3. 目标收集
4. 位移 / 地形 / 召唤 / 控制 / buff / heal / damage
5. tag linkage stateful 行为
6. 事件日志构建

#### 目标状态

效果执行统一收口到：

1. `scripts/unit_augment/unit_augment_effect_engine.gd`
   - facade，只保留公开 API 和装配

2. `scripts/unit_augment/unit_augment_effect_runtime_gateway.gd`
   - 兼容边界，承接对 `Node` / `CombatManager` / `BuffManager` 的交互

3. `scripts/domain/unit_augment/effects/**`
   - 按职责拆分的 dispatcher、summary collector、query service、ops

其中核心边界为：

1. `PassiveEffectApplier`
   - 只处理 runtime stat / modifier bundle 被动写入

2. `ActiveEffectDispatcher`
   - 只负责根据 `op` 路由
   - 不保留具体 `op` 实现

3. `EffectSummaryCollector`
   - 统一维护 `summary` 初始化、累计与事件整理

4. `TargetQueryService` / `HexSpatialService`
   - 目标收集、格子范围、邻近查询、距离与落点

5. `EffectOps/*`

## 第六章：工程治理规则

### 6.1 文件规模红线

1. `> 600` 行：必须拆分评审
2. `> 900` 行：禁止继续扩写，必须先拆
3. `> 1200` 行：视为架构风险文件，优先级高于普通功能优化
4. 禁止通过删除空行、合并多语句、压缩单行控制流等方式伪造“已降到阈值以内”

### 6.2 依赖规则

1. UI 不直连 domain service
2. Domain 不引用具体 UI 节点
3. Scene 不写业务规则
4. 兼容 facade 不得反向依赖 UI

### 6.3 UI 规则

1. 新增可见 UI 必须优先定义在 `tscn`
2. 允许代码生成行级子项，但前提是对应项本身为子场景实例
3. 禁止继续直接手写裸控件树

### 6.4 接口规则

1. 核心服务不得以 `call()` 作为常规调用手段
2. 所有 facade 必须有清晰注释说明：
   - 兼容对象是谁
   - 计划何时移除
   - 新接口对应位置在哪
3. facade 只允许保留公开 API、装配与兼容网关，不得继续承载大段业务实现

### 6.5 可读性与注释规则

1. 生产代码以 `doc/GDScript可读性与编码规范.md` 为统一标准
2. 改动文件的中文注释行 / 非空代码行默认不得低于 `20%`
3. 禁止单行函数体、单行控制流、超长函数体和过度紧凑排版
4. 行数治理的目标是模块拆分，不是排版压缩

### 6.6 测试规则

1. 核心重构必须带回归测试
2. facade 迁移必须有 contract tests
3. UI 输入优先级改动必须测商店 / tooltip / 背包 / 战场交互冲突

### 6.7 Review Checklist

每次结构性 PR 必查：

1. 是否扩大了 god class
2. 是否新增了代码拼 UI
3. 是否新增了 `_get_root_node(...)`
4. 是否新增了跨层 `call()`
5. 是否破坏了现有回归测试
6. 是否引入了新的运行时隐式装配

## 第七章：风险与回退策略

### 7.1 主要风险

1. **行为回归风险**
   - 战斗、功法、地形、buff、输入优先级都高度耦合

2. **迁移双轨风险**
   - 新旧 facade 并存期间，容易出现重复逻辑和状态漂移

3. **性能风险**
   - UI scene-first 后，如果不控制节点复用，可能引入额外创建销毁开销

4. **输入冲突风险**
   - 当前 `_input` / `_unhandled_input` 已经较复杂，分层迁移时容易打破优先级

### 7.2 回退策略

1. 每个 Phase 单独合并，不做跨阶段巨型提交
2. facade 保留一个稳定窗口，不在拆分未完成时直接删旧入口
3. 每个阶段都要求：
   - 回归测试通过
   - 可以单独回滚
   - 不依赖下一阶段才能运行

### 7.3 兼容策略

前两阶段默认保持：

1. JSON 数据 schema 不变
2. Mod 目录组织不变
3. `EventBus` 对外语义不变
4. 旧场景入口不变

1. `battlefield` 不再通过继承叠加运行时、UI、编排三层职责

1. 战斗启动 / 停止

3. 商店刷新

1. 核心回归测试

## 总结

当前项目的问题不是“需要整理一下”，而是已经形成了典型的**演化型结构债务**：系统能跑，但边界失真，主对象过载，UI 半场景化半代码化，运行时装配和动态调用成为默认手段。

这类项目不适合“一次性重写”，但也绝不能继续在原结构上温和修补。正确路径只有一条：

1. 先冻结结构继续恶化
2. 再通过 facade 和组合化把核心服务逐步抽离
3. 最后把场景、UI、应用编排、领域规则重新放回各自该在的位置

从现在开始，项目的首要任务不是继续增长功能，而是恢复结构的可信度。没有这个前提，后续所有功能都只是在债务上叠债务。
