# 项目代码重构方案

> 角色定位：本文档不是“优化建议清单”，而是当前项目的重构主设计文档。后续所有结构调整、拆分实施、阶段性验收，都应以本文档为统一依据，而不是零散口头约定。

> 重构策略：**渐进式绞杀**。保持现有项目持续可运行、可回归、可回退，在兼容旧接口和旧数据的前提下，逐步把系统迁移到新的稳定分层。

---

## 第一章：现状诊断

### 1.1 当前项目的结构性问题

当前项目的主要问题不是单个文件写得“丑”或“长”，而是**架构边界已经失控**。现状可以归纳为以下几类：

1. **战场主对象职责叠加**
   - 当前战场对象存在多层继承链：
     - `scripts/board/battlefield.gd`
     - `scripts/ui/battlefield_ui.gd`
     - `scripts/battle/battlefield_runtime.gd`
   - 这条继承链把以下职责叠在同一个对象上：
     - 战场运行时
     - 战场 UI
     - 输入路由
     - 商店 / 关卡 / 库存 / 回收编排
     - 战斗结束与阶段切换
   - 结果是：任何一个局内需求，都可能继续把代码堆进同一条继承链，进一步扩大耦合面。

2. **UI 只有一半在场景里，另一半仍在代码里现拼**
   - `scenes/battle/battlefield.tscn` 已经定义了顶栏、商店主面板、详情面板、背包面板、tooltip、备战席等大量基础结构。
   - 但脚本中仍然大量使用 `Label.new()`、`Button.new()`、`VBoxContainer.new()`、`HBoxContainer.new()` 等方式动态拼装：
     - 库存卡片
     - 商店卡片
     - 功法槽 / 装备槽行
     - 过滤按钮
     - tooltip 列表项
   - 这导致 UI 无法在编辑器中完整所见即所得，任何布局调整都必须回到脚本读代码。

3. **超大 God Object 普遍存在**
   - 当前主要高风险文件：

| 文件 | 行数 | 问题 |
|---|---:|---|
| `scripts/ui/battlefield_ui.gd` | 2054 | UI、输入、详情、背包、日志、tooltip 过载 |
| `scripts/board/battlefield.gd` | 1774 | 正式战斗编排、商店、关卡、回收、库存、战斗面板混杂 |
| `scripts/gongfa/gongfa_manager.gd` | 1738 | 装配、触发、效果、监听、日志、数据访问、状态机混杂 |
| `scripts/gongfa/effect_engine.gd` | 1484 | 被动、主动、目标查询、效果分发、战场副作用全部揉在一起 |
| `scripts/combat/combat_manager.gd` | 1274 | 注册、移动、攻击、地形、缓存、事件中继、结算全耦合 |

4. **运行时装配泛滥**
   - 代码大量在 `_ready()` 或 `_bootstrap_*()` 中执行：
     - `.new() as Node`
     - `add_child(...)`
   - 典型例子：
     - `battlefield_runtime.gd` 运行时创建 `RuntimeUnitDeployManager`、`RuntimeDragController`、`RuntimeBattlefieldRenderer`
     - `battlefield.gd` 运行时创建 `RuntimeShopController`、`RuntimeStageBridge`、`RuntimeInventoryController`、`RuntimeRecycleController`
   - 这种做法的直接后果：
     - 依赖关系不可视
     - 场景树不可读
     - 调试困难
     - 编辑器无法准确反映系统结构

5. **动态调用过多，静态边界过弱**
   - 代码广泛依赖：
     - `call(...)`
     - `_get_root_node("...")`
     - `get_node_or_null(...)`
   - 动态调用本身不是原罪，但在当前项目里已经被用作**替代模块接口设计**。
   - 结果是：
     - 类型关系模糊
     - 重构成本被隐藏
     - 调用链难以追踪
     - 局部改动容易引发级联回归

6. **目录分层与实际职责不匹配**
   - 原始规划希望有明确的 `scenes/ui/`、应用分层、组合优于继承。
   - 实际上：
     - `scripts/ui/` 中仍然夹杂大量业务逻辑
     - `scripts/board/` 和 `scripts/battle/` 的边界不清晰
     - `scripts/gongfa/` 既是数据入口、又是战斗监听器、又是效果总线
   - 目录看似分层，实际仍是横向耦合。

### 1.2 根因判断

从项目演化角度看，当前结构问题的根因不是“代码质量不好”，而是以下几项长期趋势叠加造成：

1. 以**功能迭代优先**替代了**边界维护优先**
2. 使用了“先塞进主对象，后面再拆”的模式，但“后面”长期没有真正发生
3. 把运行时动态装配当作模块化，实际上只是把耦合转移成了隐藏耦合
4. UI 已经开始场景化，但没有明确规定“禁止新增代码拼 UI”
5. 超大脚本虽已开始局部拆 controller，但拆分后的控制器仍依附在旧 God Object 下，未形成真正的新架构

### 1.3 结论

项目已经进入**必须进行底层结构重构**的阶段。继续在现有结构上迭代，会出现以下后果：

1. 每次新功能都进一步扩大 `battlefield` / `gongfa_manager` / `combat_manager` 的职责
2. UI 调整成本持续升高
3. 输入冲突、事件错连、生命周期混乱会越来越频繁
4. 测试成本会越来越高，但问题定位速度越来越慢
5. 后续任何真正的大功能都会先被结构阻力拖死

## 第二章：重构目标与原则

### 2.1 重构目标

本次重构的目标不是“清理代码风格”，而是建立一个可以长期承载项目迭代的稳定结构：

1. **场景负责声明结构**
   - 所有可编辑 UI 必须在 `tscn` 或子场景中定义。
   - 场景树必须反映真实运行结构。

2. **脚本负责绑定与协调**
   - 场景脚本只做节点绑定、状态投影、输入转发、调用应用层接口。
   - 不再承担跨系统业务编排。

3. **应用层负责用例编排**
   - 战斗启动、阶段推进、商店刷新、库存操作、回收出售等应由应用层协调。

4. **领域层负责业务规则**
   - `combat` 只处理战斗模拟。
   - `gongfa` 只处理功法和效果规则。
   - `stage`、`economy`、`inventory`、`shop` 保持独立边界。

5. **基础设施负责装配与兼容**
   - 单例访问、动态调用、兼容 facade 应收口到基础设施与过渡层。

### 2.2 设计原则

1. **组合优于继承**
   - 废弃 `battlefield -> battlefield_ui -> battlefield_runtime` 这类职责叠层。

2. **scene-first UI**
   - 可以在编辑器中看到的 UI，必须尽量在编辑器中定义。

3. **接口显式化**
   - 核心领域内部不允许继续以 `call()` 代替接口设计。

4. **渐进迁移**
   - 不进行一次性重写，不中断现有开发，不破坏当前数据和玩法。

5. **兼容优先**
   - 前两阶段不修改 JSON schema、不修改 Mod 组织、不改外部语义。

6. **可回归、可回退**
   - 每个阶段必须具备明确退出条件和回归测试覆盖。

## 第三章：目标架构蓝图

### 3.1 分层结构

目标架构采用五层：

```text
Scene Layer
  ├─ 场景声明
  ├─ 子场景组合
  └─ 节点依赖可视化

Application Layer
  ├─ 用例编排
  ├─ 阶段协调
  ├─ 输入路由
  └─ 跨模块流程控制

Domain Layer
  ├─ Combat
  ├─ Gongfa
  ├─ Stage
  ├─ Economy
  ├─ Inventory
  └─ Shop

UI Layer
  ├─ Presenter
  ├─ ViewBinder
  ├─ ViewModel Adapter
  └─ 可复用 UI 子场景

Infrastructure Layer
  ├─ Facade / Compatibility
  ├─ Service Locator / DI Entry
  ├─ EventBus Adapter
  └─ Data Access / Runtime Wiring
```

### 3.2 依赖方向

依赖方向必须固定为：

```text
Scene -> UI / App
UI -> App
App -> Domain
App -> Infra
Domain -> Domain
Infra -> Domain
```

禁止出现：

1. UI 直接访问 `CombatManager` / `GongfaManager`
2. Domain 直接操作具体 UI 节点
3. 场景层直接跨模块写业务规则
4. Domain 到处直接 `_get_root_node("...")`

### 3.3 战场子系统目标结构

战场主场景应改为组合：

```text
BattlefieldScene
  ├─ BattlefieldCoordinator
  ├─ BattlefieldWorldController
  ├─ BattlefieldHudPresenter
  ├─ BattlefieldSceneRefs
  ├─ CombatFacade
  ├─ GongfaFacade
  ├─ StageFacade
  ├─ ShopFacade
  └─ UI SubScenes
```

职责定义：

1. `BattlefieldScene`
   - 只负责场景装配、节点引用、初始化顺序

2. `BattlefieldCoordinator`
   - 负责局内用例编排
   - 例如：开始战斗、战斗结束、推进关卡、刷新商店、打开详情

3. `BattlefieldWorldController`
   - 只处理世界交互
   - 例如：拖拽部署、镜头缩放、右键平移、战场单位点击

4. `BattlefieldHudPresenter`
   - 只处理 HUD、日志、详情、背包、商店数据投影

5. `BattlefieldSceneRefs`
   - 统一维护场景节点引用
   - 让其他层通过显式依赖拿到节点，而不是四处写路径

## 第四章：重点子系统拆分方案

### 4.1 战场场景重构

#### 当前问题

1. 继承链承载了运行时、UI、正式局内编排三层职责
2. controller 和 manager 在运行时动态创建，导致场景树不可视
3. 输入优先级散落在多个类里
4. 商店、库存、回收、日志、详情等局内 UI 直接挂在战场根脚本上

#### 目标方案

1. 取消 `battlefield.gd -> battlefield_ui.gd -> battlefield_runtime.gd` 继承链
2. 新战场根脚本只做装配
3. 将输入、UI、编排分为独立组合对象
4. 所有运行时控制器改为：
   - 场景内显式子节点，或
   - 由装配器统一注入，但必须在结构上可追踪

#### 迁移边界

第一阶段不直接推翻 `Battlefield` 玩法，只先引入新组合骨架和 facade，让旧流程还能走通。

### 4.2 战斗核心拆分

#### 当前问题

`combat_manager.gd` 当前同时负责：

1. 单位注册
2. 战斗主循环
3. 移动
4. 攻击
5. 地形
6. 缓存
7. 事件中继
8. 结算
9. 运行时 metrics

#### 目标拆分

拆为以下服务：

1. `CombatSimulation`
   - 逻辑帧推进
   - 战斗阶段驱动

2. `CombatUnitRegistry`
   - 战场单位注册
   - 存活单位查询
   - 队伍计数

3. `CombatMovementService`
   - cell 占用
   - 强制移动
   - 寻路移动
   - 移动失败/成功上报

4. `CombatAttackResolver`
   - 选敌
   - 射程判断
   - 攻击尝试
   - 伤害结算回调

5. `CombatTerrainService`
   - 临时地形
   - 静态地形
   - 地形标签
   - 阻挡与可视更新

6. `CombatEventRelay`
   - Godot signal 发射
   - 外部战斗事件兼容
   - battle summary 汇总

#### 兼容策略

保留 `CombatManager` 作为 facade，在第一阶段继续对上层暴露现有方法名，内部逐步转发到新服务。

### 4.3 功法系统拆分

#### 当前问题

`gongfa_manager.gd` 当前同时承担：

1. 数据重载
2. 装备 / 功法配置
3. 被动效果应用
4. 自动触发轮询
5. 技能释放
6. 事件监听
7. tag linkage
8. buff 生命周期联动
9. 日志事件发射

#### 目标拆分

拆为以下服务：

1. `GongfaLoadoutService`
   - 功法 / 装备装载与卸载
   - runtime loadout 状态

2. `GongfaTriggerService`
   - 触发器构建
   - 轮询判断
   - trigger entry 生命周期

3. `GongfaEffectContextBuilder`
   - 统一生成 effect context
   - 统一构建 battle / target / event / buff 相关上下文

4. `GongfaBattleListener`
   - 监听 combat 事件
   - 将事件映射为功法触发

5. `GongfaTelemetryEmitter`
   - 统一处理 skill_triggered / buff_event / heal / damage 日志

#### 兼容策略

保留 `GongfaManager` facade，逐步把内部状态迁移给以上服务。

### 4.4 效果系统拆分

#### 当前问题

`effect_engine.gd` 已经成为“效果总宇宙”：

1. 被动效果处理
2. 主动效果分发
3. 目标收集
4. 位移 / 地形 / 召唤 / 控制 / buff / heal / damage
5. tag linkage stateful 行为
6. 事件日志构建

#### 目标拆分

拆为以下部分：

1. `PassiveEffectApplier`
   - 只处理 runtime stat / modifier bundle 被动写入

2. `ActiveEffectDispatcher`
   - 只负责根据 `op` 分发

3. `TargetQueryService`
   - 收集敌人、友军、最近单位、低血单位、格子范围

4. `EffectOps/*`
   - 按能力拆文件，例如：
     - `effect_ops_damage.gd`
     - `effect_ops_heal.gd`
     - `effect_ops_buff.gd`
     - `effect_ops_movement.gd`
     - `effect_ops_terrain.gd`
     - `effect_ops_summon.gd`
     - `effect_ops_control.gd`
     - `effect_ops_tag_linkage.gd`

#### 兼容策略

保留 `EffectEngine` facade，旧入口仍是：

1. `apply_passive_effects(...)`
2. `execute_active_effects(...)`

内部改由 dispatcher 和 op services 执行。

### 4.5 UI 展示层重构

#### 当前问题

`battlefield_ui.gd` 中仍存在大量代码拼 UI：

1. 过滤按钮
2. 库存卡片
3. 商店卡片
4. 功法槽行
5. 装备槽行
6. tooltip 列表项

#### 目标方案

新建 `scenes/ui/` 子场景库：

1. `inventory_item_card.tscn`
2. `shop_offer_card.tscn`
3. `gongfa_slot_row.tscn`
4. `equipment_slot_row.tscn`
5. `battle_log_panel.tscn`
6. `unit_detail_panel.tscn`
7. `item_tooltip.tscn`
8. `unit_tooltip.tscn`

脚本约束：

1. UI 子场景脚本只暴露：
   - `setup(view_model)`
   - `refresh(view_model)`
   - `bind(actions)`
2. UI 脚本禁止直连 `GongfaManager` / `CombatManager`
3. UI 只能消费由 presenter 提供的数据

### 4.6 单例访问与动态调用收口

#### 当前问题

项目中大量存在：

1. `_get_root_node("EventBus")`
2. `_get_root_node("DataManager")`
3. `_get_root_node("GameManager")`
4. `call("...")`

#### 目标方案

1. 引入 `ServiceLocator` 或等价的依赖入口
2. 场景装配期显式注入常用服务
3. `call()` 仅允许保留在：
   - 过渡 facade
   - editor / debug / adapter 边界
4. 核心 domain 服务之间使用显式方法签名

## 第五章：迁移路线图

### Phase 0：冻结结构与规则

#### 目标

建立架构红线，停止结构继续恶化。

#### 必做项

1. 新增本文档
2. 新增工程规则：
   - 禁止新增超长 god class
   - 禁止新增大块代码拼 UI
   - 禁止在核心层新增 `_get_root_node(...)`
3. 定义目标目录：
   - `scripts/app/`
   - `scripts/domain/`
   - `scripts/ui/`
   - `scripts/infra/`
   - 或语义等价目录

#### 退出条件

1. 团队对重构边界口径统一
2. 后续新改动不再继续污染旧结构

### Phase 1：战斗 / 功法核心解耦

#### 目标

优先拆 `combat_manager.gd`、`gongfa_manager.gd`、`effect_engine.gd`，但不改玩法和数据外观。

#### 必做项

1. 引入 facade 兼容层
2. 拆出 combat 子服务
3. 拆出 gongfa 子服务
4. 拆出 effect dispatcher 和 op services
5. 把监听、上下文、日志、查询从 God Object 剥离

#### 禁止项

1. 不允许一边拆一边继续把新功能加回旧 God Object
2. 不允许在新服务里继续复制旧的 `call()` 风格

#### 退出条件

1. `combat_manager.gd` 降为 facade + 装配器
2. `gongfa_manager.gd` 降为 facade + 协调器
3. `effect_engine.gd` 降为 dispatcher facade
4. 现有战斗与功法核心回归测试通过

### Phase 2：战场主场景改为组合编排

#### 目标

替换战场继承链，建立组合式局内外壳。

#### 必做项

1. 新建 `BattlefieldScene`
2. 新建 `BattlefieldCoordinator`
3. 新建 `BattlefieldWorldController`
4. 新建 `BattlefieldHudPresenter`
5. 逐步迁移旧 `battlefield_*` 脚本职责

#### 退出条件

1. 战场根场景不再通过多层 `extends` 混合职责
2. 输入、编排、UI 展示职责分开

### Phase 3：UI 全量 scene-first

#### 目标

把所有大块动态 UI 拼装迁移到子场景。

#### 必做项

1. 建立 `scenes/ui/` 子场景库
2. 迁移库存、商店、详情、槽位、tooltip
3. 建立 presenter -> subscene 的渲染接口

#### 退出条件

1. 新增 UI 元素必须有子场景
2. `battlefield_ui.gd` 中的大量 `Control.new()` 被移除

### Phase 4：统一依赖与模块边界

#### 目标

清理所有遗留的隐式依赖与跨层访问。

#### 必做项

1. 收口单例访问
2. 清理根节点查找
3. 清理跨层 `call()`
4. 将 main / stage / economy / inventory / shop 纳入同一应用层规范

#### 退出条件

1. 依赖方向可静态描述
2. 旧兼容层明显收缩

## 第六章：工程治理规则

### 6.1 文件规模红线

1. `> 600` 行：必须拆分评审
2. `> 900` 行：禁止继续扩写，必须先拆
3. `> 1200` 行：视为架构风险文件，优先级高于普通功能优化

### 6.2 依赖规则

1. UI 不直连 domain service
2. Domain 不引用具体 UI 节点
3. Scene 不写业务规则
4. 兼容 facade 不得反向依赖 UI

### 6.3 UI 规则

1. 新增可见 UI 必须优先定义在 `tscn`
2. 允许代码生成行级子项，但前提是对应项本身为子场景实例
3. 禁止继续直接手写裸控件树

### 6.4 接口规则

1. 核心服务不得以 `call()` 作为常规调用手段
2. 所有 facade 必须有清晰注释说明：
   - 兼容对象是谁
   - 计划何时移除
   - 新接口对应位置在哪

### 6.5 测试规则

1. 核心重构必须带回归测试
2. facade 迁移必须有 contract tests
3. UI 输入优先级改动必须测商店 / tooltip / 背包 / 战场交互冲突

### 6.6 Review Checklist

每次结构性 PR 必查：

1. 是否扩大了 god class
2. 是否新增了代码拼 UI
3. 是否新增了 `_get_root_node(...)`
4. 是否新增了跨层 `call()`
5. 是否破坏了现有回归测试
6. 是否引入了新的运行时隐式装配

---

## 第七章：风险与回退策略

### 7.1 主要风险

1. **行为回归风险**
   - 战斗、功法、地形、buff、输入优先级都高度耦合

2. **迁移双轨风险**
   - 新旧 facade 并存期间，容易出现重复逻辑和状态漂移

3. **性能风险**
   - UI scene-first 后，如果不控制节点复用，可能引入额外创建销毁开销

4. **输入冲突风险**
   - 当前 `_input` / `_unhandled_input` 已经较复杂，分层迁移时容易打破优先级

### 7.2 回退策略

1. 每个 Phase 单独合并，不做跨阶段巨型提交
2. facade 保留一个稳定窗口，不在拆分未完成时直接删旧入口
3. 每个阶段都要求：
   - 回归测试通过
   - 可以单独回滚
   - 不依赖下一阶段才能运行

### 7.3 兼容策略

前两阶段默认保持：

1. JSON 数据 schema 不变
2. Mod 目录组织不变
3. `EventBus` 对外语义不变
4. 旧场景入口不变

## 第八章：验收标准

### 8.1 结构验收

以下条件全部满足，才视为重构进入有效状态：

1. `battlefield` 不再通过继承叠加运行时、UI、编排三层职责
2. `combat_manager.gd`、`gongfa_manager.gd`、`effect_engine.gd` 降为 facade 或协调器
3. 新增 UI 元素存在对应 `tscn` 子场景
4. 运行时 `new()+add_child()` 的控制器装配被压缩到少数装配边界

### 8.2 工程指标

建议阶段性指标：

1. 超过 900 行的业务脚本数量逐步降到 0
2. 超过 600 行的高风险脚本数量显著下降
3. `battlefield_ui.gd` 中裸 `Control.new()` 类代码逐步归零
4. 核心业务层对 `_get_root_node(...)` 的依赖逐步归零

### 8.3 行为验收

以下行为必须与当前版本一致：

1. 战斗启动 / 停止
2. 拖拽部署
3. 商店刷新
4. 库存显示与装卸
5. 单位详情
6. 战斗日志
7. 功法触发
8. 地形效果
9. buff / debuff 生命周期
10. 输入优先级

### 8.4 测试验收

必须具备以下测试类型：

1. 核心回归测试
2. facade contract tests
3. 静态结构检查
4. 输入交互回归
5. 性能基线检查

### 8.5 最终目标状态

重构完成后的项目应该满足：

1. 任何 UI 调整都优先在编辑器完成，而不是先改脚本
2. 任何战斗或功法规则改动都能定位到明确子服务，而不是去 God Object 中翻找
3. 任何新功能都能找到清晰归属层，而不是“先塞到 battlefield 再说”
4. 项目结构对新开发者可解释，而不是只能靠历史上下文猜测

## 总结

当前项目的问题不是“需要整理一下”，而是已经形成了典型的**演化型结构债务**：系统能跑，但边界失真，主对象过载，UI 半场景化半代码化，运行时装配和动态调用成为默认手段。

这类项目不适合“一次性重写”，但也绝不能继续在原结构上温和修补。正确路径只有一条：

1. 先冻结结构继续恶化
2. 再通过 facade 和组合化把核心服务逐步抽离
3. 最后把场景、UI、应用编排、领域规则重新放回各自该在的位置

从现在开始，项目的首要任务不是继续增长功能，而是恢复结构的可信度。没有这个前提，后续所有功能都只是在债务上叠债务。
