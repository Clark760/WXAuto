# M5 战斗数据统计修正（面向代码 AI）

> **目标读者**：AI 代码生成器。
> **涉及文件**：`battle_statistics.gd`、`battle_flow.gd`、`unit_combat.gd`、`gongfa_manager.gd`、`buff_manager.gd`、`combat_manager.gd`、`battle_stats_panel.gd`。
> **核心问题**：当前统计系统存在多个伤害/治疗渠道未被记录的遗漏，且承伤统计无法区分护盾/无敌吸收的伤害。

---

## 一、当前统计架构

### 1.1 数据流总览

```
普攻伤害：
  CombatManager._try_execute_attack
    → UnitCombat.try_attack_target → UnitCombat.receive_damage
    → CombatManager.damage_resolved 信号
    → BattleFlow._on_damage_resolved → BattleStatistics.record_damage ✅

功法/装备技能伤害：
  GongfaManager._try_fire_skill
    → EffectEngine.execute_active_effects
    → GongfaManager._emit_effect_log_events → skill_effect_damage 信号
    → BattleFlow._on_skill_effect_damage → BattleStatistics.record_damage ✅

Buff/Debuff 持续伤害（DOT）：
  BuffManager.tick → tick_requests
    → GongfaManager._execute_buff_tick_requests
    → EffectEngine.execute_active_effects
    → GongfaManager._emit_effect_log_events → skill_effect_damage 信号
    → BattleFlow._on_skill_effect_damage → BattleStatistics.record_damage ✅

战场效果（hazard_zone / 临时地形 tick）：
  BuffManager._battlefield_effects tick → tick_requests（event_type: "battlefield_tick"）
    → 同上 Buff tick 路径 ✅

环境伤害（Boss 机制等）：
  CombatManager.apply_environment_damage
    → UnitCombat.receive_damage
    → CombatManager.damage_resolved 信号（is_environment=true）
    → BattleFlow._on_damage_resolved → BattleStatistics.record_damage ✅
```

### 1.2 统计键（`battle_statistics.gd` 第 13-19 行）

| 键 | 含义 | 当前录入来源 |
|----|------|-------------|
| `damage_dealt` | 造成伤害 | 普攻 + 技能 + DOT + 环境 |
| `damage_taken` | 承受伤害 | 同上（目标侧） |
| `healing_done` | 治疗量 | 仅 `skill_effect_heal` 信号 |
| `kills` | 击杀 | `unit_died` 信号 |
| `deaths` | 死亡 | `unit_died` 信号 |

## 二、已确认的遗漏（7 项）

### 遗漏 1：反伤（Thorns）伤害 — 未记录

**位置**：`unit_combat.gd` → `_apply_thorns_reflect`（第 551-571 行）

**问题**：反伤直接调用 `source_combat.call("receive_damage", ...)` 但不经过 `CombatManager`，因此 `damage_resolved` 信号不会触发。反伤伤害既不计入反伤者的 `damage_dealt`，也不计入被反伤者的 `damage_taken`。

**修复方案**：

```gdscript
# 方案 A（推荐）：在 _apply_thorns_reflect 结束后，手动 emit 一个信号
# 需要在 UnitCombat 新增信号：
signal thorns_damage_dealt(source: Node, target: Node, damage: float)

# 在 _apply_thorns_reflect 末尾（第 571 行前）添加：
thorns_damage_dealt.emit(owner_unit, source, reflect_damage)

# 在 BattleFlow._connect_signals 中监听该信号，调用 record_damage。
```

```gdscript
# 方案 B：改为通过 CombatManager.apply_environment_damage 施加反伤
# 这样自动走 damage_resolved 路径，但需要传入反伤来源作为 source。
```

---

### 遗漏 2：生命偷取（Vampire）治疗 — 未记入 `healing_done`

**位置**：`unit_combat.gd` → `_apply_vampire_lifesteal`（第 542-548 行）

**问题**：吸血直接调 `restore_hp`，不触发任何统计信号。吸血回复的 HP 不计入攻击者的 `healing_done`。

**修复方案**：

```gdscript
# 在 UnitCombat 新增信号：
signal healing_performed(source: Node, target: Node, amount: float, heal_type: String)

# 在 _apply_vampire_lifesteal 的 restore_hp 调用后添加：
healing_performed.emit(owner_unit, owner_unit, dealt_damage * vampire_ratio, "vampire")

# 在 BattleFlow 中监听，调用 record_healing。
```

---

### 遗漏 3：被动 HP 回复（hp_regen_add / passive_hp_regen） — 未记入 `healing_done`

**位置**：`unit_combat.gd` → `tick_logic`（第 90-97 行）

**问题**：每秒自然回复直接调 `restore_hp`，不触发统计。功法/装备的 `hp_regen_add` 导致的所有回复均未记录。

**修复方案**：

```gdscript
# 在 tick_logic 中，restore_hp 调用后：
var actual_regen: float = hp_regen_per_sec * delta
if actual_regen > 0.0:
    restore_hp(actual_regen)
    healing_performed.emit(owner_unit, owner_unit, actual_regen, "regen")
```

> ⚠️ **注意**：HP 回复是高频事件（每逻辑帧触发），若全部录入会造成统计信号过于密集。建议设置累计阈值（如每累积 50 点回复才统计一次），或在 `BattleStatistics` 中直接批量累加，不发信号。

### 遗漏 4：护盾吸收 — 未独立统计

**位置**：`unit_combat.gd` → `receive_damage`（第 217-240 行）

**问题**：护盾吸收的伤害直接返回 `damage: 0`（被护盾完全挡住时），或返回护盾之后的剩余伤害。`battle_statistics.gd` 的 `record_damage` 仅看 `damage` 字段，所以：
- 被护盾完全挡住的伤害 → **攻击方 damage_dealt = 0，目标 damage_taken = 0**
- 被护盾部分挡住的伤害 → **仅记录穿盾后的伤害，吸收部分丢失**

**影响**：

| 场景 | 攻击者视角 | 目标视角 |
|------|-----------|---------|
| 100 伤害被护盾完全挡住 | damage_dealt += 0 ❌ | damage_taken += 0 ❌ |
| 100 伤害护盾挡 60，穿透 40 | damage_dealt += 40 ⚠️ | damage_taken += 40 ⚠️ |

**修复方案**：

```gdscript
# 方案 A（推荐）：在 BattleStatistics 新增统计键
const STAT_KEYS: Array[String] = [
    "damage_dealt",          # 造成实际伤害（穿盾后）
    "damage_dealt_total",    # 造成总伤害（含被护盾吸收）← 新增
    "damage_taken",          # 承受实际伤害（穿盾后）
    "damage_taken_total",    # 承受总伤害（含护盾吸收）← 新增
    "shield_absorbed",       # 护盾吸收总量 ← 新增
    "healing_done",
    "kills",
    "deaths"
]

# 在 BattleFlow._on_damage_resolved 中：
var shield_absorbed: int = int(round(float(event_dict.get("shield_absorbed", 0.0))))
var total_damage: int = damage_value + shield_absorbed
_statistics.call("record_damage", source_unit, target_unit, damage_value)
# 额外记录总伤害和护盾吸收：
_statistics.call("record_stat", source_unit, "damage_dealt_total", total_damage)
_statistics.call("record_stat", target_unit, "damage_taken_total", total_damage)
_statistics.call("record_stat", target_unit, "shield_absorbed", shield_absorbed)
```

### 遗漏 5：无敌/免疫伤害 — 未记录

**位置**：`unit_combat.gd` → `receive_damage`（第 242-254 行）

**问题**：`stage_damage_immune = true` 时直接返回 `damage: 0`，攻击方的输出不计入统计。

**修复方案（与护盾合并）**：

```gdscript
# 在 record_damage 的同时，也记录 damage_dealt_total（原始意图伤害）：
# 免疫返回时在事件中增加一个 "immune_absorbed" 标记。

# 在 receive_damage 的免疫分支中添加：
immune_event["immune_absorbed"] = maxf(amount, 0.0)

# 在 BattleFlow 中捕获并记录：
var immune_absorbed: int = int(round(float(event_dict.get("immune_absorbed", 0.0))))
if immune_absorbed > 0:
    _statistics.call("record_stat", source_unit, "damage_dealt_total", immune_absorbed)
```

### 遗漏 6：Buff tick 造成伤害时 source 可能为 null

**位置**：`gongfa_manager.gd` → `_execute_buff_tick_requests`（第 627-667 行）

**问题**：`source` 从 `_unit_lookup` 按 `source_id` 查找，若施法者已死亡或被移除，source = null。`_emit_effect_log_events` 仍会 emit `skill_effect_damage`，但 `BattleFlow` 中 `record_damage(null, target, amount)` 时 `register_unit(null)` 返回 -1，导致 **source 的 damage_dealt 不被记录**。

> 目标侧 damage_taken 正常记录（target 仍存活）。

**修复方案**：

```gdscript
# 选项 1：即使 source 为 null，仍记录 target 的 damage_taken（当前已做到）。
# 选项 2（推荐）：在 buff 施加时记录 source_unit_name / source_unit_id，
#   即使 source 节点失效，也能通过备选信息创建"幽灵统计行"。
#   在 battle_statistics.gd 的 record_damage 中：
func record_damage(source: Node, target: Node, amount: float,
                   source_fallback: Dictionary = {}) -> void:
    ...
    if source_iid <= 0 and not source_fallback.is_empty():
        source_iid = _register_fallback(source_fallback)
    ...
```

### 遗漏 7：Buff tick 造成治疗时未记入 `healing_done`

**位置**：`gongfa_manager.gd` → `_emit_effect_log_events`（第 706-727 行）

**问题**：虽然 `heal_events` 会通过 `skill_effect_heal.emit` 触发并在 `BattleFlow._on_skill_effect_heal` 中调用 `record_healing` ✅，但需要确认 EffectEngine 对 `heal_self` tick op 正确生成 `heal_events`。

**验证结论**：经检查 EffectEngine 代码，`heal_self` 操作确实写入 `summary["heal_events"]`。这条路径 **已正确录入**。此项为假阳性。

## 三、承伤统计——护盾/无敌的处理

### 3.1 当前行为

| 防御机制 | `damage_taken` 记录值 | 问题 |
|---------|---------------------|------|
| **无防御** | `final_damage`（实际扣血量） | ✅ 正确 |
| **护盾完全挡住** | `0` | ❌ 承伤看起来为 0 |
| **护盾部分挡住** | 穿盾后的伤害 | ⚠️ 承伤偏低 |
| **无敌（stage_damage_immune）** | `0` | ❌ 承伤看起来为 0 |
| **减伤（damage_reduce_percent）** | 减伤后的值 | ✅ 正确（反映实际承伤） |
| **闪避** | `0` | ✅ 正确（闪避 = 未命中） |

### 3.2 建议承伤分层

```
damage_taken          现有键，记录"实际扣血量"（穿盾+减伤后）
damage_taken_total    新增键，记录"命中伤害意图"（穿盾前、减伤前、但闪避除外）
shield_absorbed       新增键，记录"护盾吸收总量"
damage_immune_blocked 新增键，记录"无敌/免疫挡掉的伤害"
```

UI 面板中"承伤排行"应使用 `damage_taken_total`（更能反映谁在前排扛伤），同时在角色详情悬浮框中展示分项明细。
