# M5 角色移动逻辑修正

> **目标读者**：AI 代码生成器。本文档列出角色战斗移动的所有已知问题及修改建议。
> **涉及文件**：`scripts/combat/combat_manager.gd`

---

## 一、核心 Bug：攻击范围内有目标但仍移动

### 现状

`_run_unit_logic()`（行 627-720）的决策流程：

```
1. pick target
2. if allow_attack → _try_execute_attack(unit, combat, target)
   └─ 成功 → return（不移动）✓
   └─ 失败 → 继续往下走...
3. if allow_move → 选邻格 → 执行移动
```

`_try_execute_attack()`（行 1329-1346）在以下场景返回 `false`：

| 场景 | 返回值 | 是否应该移动？ |
|------|--------|--------------|
| target == null | false | ✓ 应该移动去找 |
| 目标**不在攻击范围**内 | false | ✓ 应该移动靠近 |
| 目标在范围内但 **CD 没好** | false | ❌ **不应移动，应原地等** |
| 目标在范围内但 `performed=false`（其他原因） | false | ❌ **不应移动** |

**根因**：`_try_execute_attack` 向上层返回的是单一 `bool`，调用方无法区分"不在射程"和"在射程但 CD 没好"这两种截然不同的情况。

### 修改建议

**方案 A（推荐 — 最小改动）**：在 `_run_unit_logic()` 的移动分支前，额外检查"目标是否已在攻击范围内"：

```gdscript
# --- 行 644-650 之后、行 652（移动分支）之前插入 ---

# 核心修复：目标已在攻击范围内时，即使不能攻击也原地待命。
if target != null and _is_target_in_attack_range(unit, target):
    # 在射程内：原地待命等 CD / 等攻击阶段
    var idle_movement: Node = _get_movement(unit)
    if idle_movement != null:
        idle_movement.call("clear_target")
    unit.call("play_anim_state", 0, {}) # IDLE
    return
```

**方案 B（重构方案）**：将 `_try_execute_attack` 改为返回 enum（`ATTACK_OK` / `TARGET_OUT_OF_RANGE` / `COOLDOWN_NOT_READY` / `NO_TARGET`），调用方根据结果决定行为。此方案更规范但改动更大。

## 二、`split_attack_move_phase` 导致纯移动帧无射程检查

### 现状

当 `split_attack_move_phase = true` 时（行 360-363），逻辑帧交替执行攻击帧和移动帧。在**纯移动帧**中，`allow_attack = false`，直接跳到移动分支——**完全不检查目标是否已在射程内**。

这意味着：正在射程内攻击的角色，每隔一帧就会多走一格。

### 修改建议

在移动分支的最前面（行 652 之后），无论 `allow_attack` 是否为 true，都加入射程守卫：

```gdscript
if not allow_move:
    return

# 无论是否为攻击帧，移动前都检查射程。
if target != null and _is_target_in_attack_range(unit, target):
    var idle_movement2: Node = _get_movement(unit)
    if idle_movement2 != null:
        idle_movement2.call("clear_target")
    unit.call("play_anim_state", 0, {}) # IDLE
    return
```

## 三、`allow_equal_cost_side_step` 导致无意义横移

### 现状

`_pick_best_adjacent_cell()`（行 1068-1141）中，当所有下坡邻格都被占用，但存在**同成本**邻格时，`allow_equal_cost_side_step`（默认为 true）会选择一个同成本的格子横移。

问题：当角色已经到达最优位置（流场成本等于或接近 0，即距离目标已经很近），横移没有任何战术价值，只会造成无意义的来回晃动。

### 修改建议

增加条件：只有当流场成本 > 某个阈值时才允许同成本横移：

```gdscript
# _pick_best_adjacent_cell 中，行 1121：
if allow_equal_cost_side_step and neighbor_cost == current_cost:
    # 新增守卫：低成本区域不做横移（以避免已到位的角色来回晃动）
    if current_cost <= 2:
        continue
    if not side_step_found or neighbor_focus_dist < side_step_focus_dist:
        side_step_found = true
        side_step_focus_dist = neighbor_focus_dist
        side_step_cell = neighbor
    continue
```

或直接将 `allow_equal_cost_side_step` 默认改为 `false`。

## 四、`allow_uphill_escape_step` 导致角色远离目标

### 现状

`_pick_best_adjacent_cell()`（行 1127-1134）中，当没有更优格也没有同成本格时，`allow_uphill_escape_step`（行 1139-1140）会选择一个**成本更高**（离目标更远）的格子。

这在拥堵场景下可以帮助解围，但**也会导致已经到位的角色反复远离再靠近**，表现为抖动。

### 修改建议

限制上坡步的触发条件：只有当角色周围的所有邻格都被友方占据时（真正卡死）才触发：

```gdscript
# 行 1139-1140 修改为：
if allow_uphill_escape_step and uphill_found:
    # 只在真正被完全堵住（无下坡/无同成本可走）时才上坡
    # 且 current_cost 已经较高时才触发（否则已在最优位置不必逃逸）
    if current_cost > 3:
        return uphill_cell
```

## 五、GongfaManager 技能射程未纳入"在射程内"判定

### 现状

`_is_target_in_attack_range()`（行 835-846）只检查 `combat.get_attack_range_cells()`，这是角色的**普攻范围**。

但部分角色可能有射程更远的功法技能（如远程 AOE），当前逻辑不会把技能射程纳入考虑。这意味着一个拥有 6 格远程技能但 1 格普攻的角色，在敌人距离 3 格时会继续移动靠近，而不是原地释放技能。

### 修改建议

在 `unit_combat.gd` 中新增 `get_max_effective_range_cells()` 方法，返回 `max(普攻范围, 当前可用技能最大射程)`。修改 `_is_target_in_attack_range()` 使用此方法。

```gdscript
# unit_combat.gd 新增：
func get_max_effective_range_cells() -> int:
    var base_range: int = get_attack_range_cells()
    # 如果有功法管理器，查询当前可释放的最大技能射程
    # （需要 GongfaManager 暴露接口）
    return base_range  # TODO: 集成技能射程

# combat_manager.gd 修改 _is_target_in_attack_range：
func _is_target_in_attack_range(attacker: Node, target: Node) -> bool:
    var combat: Node = _get_combat(attacker)
    if combat == null:
        return false
    var attacker_cell: Vector2i = _get_unit_cell(attacker)
    var target_cell: Vector2i = _get_unit_cell(target)
    if attacker_cell.x < 0 or target_cell.x < 0:
        return false
    var hex_dist: int = _hex_distance(attacker_cell, target_cell)
    # 使用最大有效射程（含技能）而非仅普攻射程
    var range_cells: int = maxi(int(combat.call("get_max_effective_range_cells")), 1)
    return hex_dist <= range_cells
```

> [!NOTE]
> 此项为可选优化，非必要修复。当前阶段可暂不实现技能射程集成。

## 七、合并后的 `_run_unit_logic` 修改（§一 + §二 合并）

下面给出行 627-720 的完整修正版本，仅展示差异关键区域：

```gdscript
func _run_unit_logic(unit: Node, delta: float, allow_attack: bool = true, allow_move: bool = true) -> void:
    # ... 前置检查不变 (行 628-640) ...

    combat.call("tick_logic", delta)
    var target: Node = _pick_target_for_unit(unit)

    if allow_attack:
        _metric_tick_attack_checks += 1
        _metric_total_attack_checks += 1
        if _try_execute_attack(unit, combat, target):
            _metric_tick_attacks_performed += 1
            _metric_total_attacks_performed += 1
            return

    if not allow_move:
        return

    # ========== 核心修复 ==========
    # 目标已在攻击范围内时，即使不能攻击（CD未好/攻击阶段禁用）也原地待命。
    if target != null and _is_target_in_attack_range(unit, target):
        var idle_movement: Node = _get_movement(unit)
        if idle_movement != null:
            idle_movement.call("clear_target")
        unit.call("play_anim_state", 0, {}) # IDLE
        return
    # ================================

    _metric_tick_move_checks += 1
    _metric_total_move_checks += 1
    # ... 后续移动逻辑不变 (行 657-720) ...
```

## 八、验证方式

1. 部署一个近战角色（RNG=1）面对一个静止敌人
   - **预期**：角色走到敌人旁边后停住，攻击，不再移动
   - **Bug 表现**：角色攻击后每隔一帧向前多走一格
2. 部署一个远程角色（RNG=4）面对远处敌人
   - **预期**：当敌人进入 4 格范围后角色停下射击
   - **Bug 表现**：角色在敌人 4 格范围内仍持续靠近至 1 格
3. 观察战斗整体，特别是存活角色少时的 1v1 场景
   - **预期**：双方站定对攻
   - **Bug 表现**：角色在攻击间隙来回移动
