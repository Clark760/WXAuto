# 架构约束执行细则

> 本文档是 Phase 0 的操作手册，用于把 [项目代码重构方案](./项目代码重构方案.md) 中的架构红线落实为可执行规则。后续如无明确豁免，不允许绕开本文档直接继续向旧结构加债。

> 代码可读性、中文注释和 GDScript 风格的具体要求，统一以 [GDScript可读性与编码规范](./GDScript可读性与编码规范.md) 为准。架构约束脚本会按该规范执行门禁，而不是只检查文件行数。

## 1. 目的

Phase 0 不做玩法重写，也不做大规模代码迁移。它只做两件事：

1. 冻结当前已经存在的结构债务，阻止继续恶化。
2. 为 Phase 1 之后的拆分提供统一入口、目录落点和检查机制。

## 2. 六条红线

### 2.1 禁止新增超长 God Object

适用范围：

- `scripts/**/*.gd`
- 排除 `scripts/tests/**`

执行规则：

1. 新文件超过 `600` 行，直接视为违规。
2. 已经超过 `600` 行的存量文件，禁止继续增长。
3. 已经超过 `900` 行的存量文件，只允许持平或下降，不允许净增长。

### 2.2 禁止新增脚本拼整块 UI

适用范围：

- `scripts/main/**`
- `scripts/ui/**`
- `scripts/board/**`

执行规则：

1. 新 UI 结构必须先落到 `scenes/ui/` 或对应子场景。
2. UI 脚本只允许做绑定、刷新、事件转发，不允许继续通过 `Control.new()` 一类方式拼整块控件树。
3. 现有存量动态 UI 构造按基线冻结，后续只能减少，不能增加。

### 2.3 禁止新增 `_get_root_node(...)`

适用范围：

- `scripts/**/*.gd`
- 排除 `scripts/tests/**`

执行规则：

1. Phase 0 只冻结现有旧战场链路中的存量访问。
2. 所有新出现的 `_get_root_node(...)` 都视为违规。
3. 存量文件中的该模式只允许减少，不允许增加。

### 2.4 禁止在非兼容边界新增 `call()`

适用范围：

- `scripts/**/*.gd`
- 排除 `scripts/tests/**`
- 排除 `tools/baselines/architecture_guard_exceptions.json` 中显式豁免的路径

执行规则：

1. 默认禁止新增 `call(...)` 次数。
2. 只有过渡 facade、adapter、兼容层，才允许申请豁免。
3. 豁免必须先写入 exceptions 文件，再提交代码。

### 2.5 禁止以牺牲可读性的方式压缩代码

适用范围：

- `scripts/**/*.gd`
- 排除 `scripts/tests/**`
- 对“改动文件”强制执行

执行规则：

1. 改动文件必须符合 [GDScript可读性与编码规范](./GDScript可读性与编码规范.md)。
2. 中文注释行 / 非空代码行必须满足文件级比例要求，默认不低于 `20%`。
3. 禁止单行函数体、单行控制流、超长单行、超长函数体。
4. 大文件必须保留足够空行与结构分段，不允许通过删除空行和注释来压缩行数。
5. 中文注释必须优先分布在函数入口、关键参数和复杂分支附近，不能主要集中在文件头部的 `文件说明` / `维护约束` 块里。
6. 中文注释率必须主要由函数级注释和参数边界说明承担，不能靠文件头说明或维护约束集中冲高。
7. `文件说明` / `维护约束` 注释块每块不得超过 `6` 行；超过即视为“用文件头凑注释率”。
8. 对包含 `effect`、`context`、`summary`、`runtime_gateway`、`source`、`target` 等关键参数的生产函数，注释必须在函数附近解释这些参数里至少一个的职责边界。
9. 下列行为一律视为违规，而不是“优化排版”：
   - 删除说明性注释以降低文件行数
   - 把多条语句合并到同一行
   - 把 `if/for/while/match/func` 压成单行
   - 保留同样的职责规模，只靠排版压缩到阈值以内
   - 用大段重复的文件头注释替代函数级或参数级说明

### 2.6 禁止触碰超长文件却不做模块拆分

适用范围：

- `scripts/**/*.gd`
- 排除 `scripts/tests/**`
- 对“改动文件”强制执行

执行规则：

1. 任何被修改的 `>900` 行文件，必须在同一轮改动中完成模块拆分。
2. 模块拆分的判定标准是“职责迁移到新文件/新服务”，不是“改写成更紧凑的排版”。
3. `601-900` 行文件属于高风险文件，允许桥接性维护，但必须以“继续拆分”为目标，不得新增横切职责。
4. Facade 文件只允许保留公开 API、装配逻辑、兼容网关，不允许继续承载大段业务分支。
5. `effect_engine.gd` 这类已发生过“压缩过线”问题的文件，后续再被修改时，必须优先做模块迁移与可读性修复，不得重复使用压缩手法。
6. 如果某个旧文件已经被实施细则明确列为“拆出替代服务后删除”的遗留壳，可以按文件级写入 `readability_guard` / `module_split_guard` 豁免。
7. 这种豁免只允许用于“同轮正在落地替代实现”的旧壳文件，不能拿来给继续存活的 God Object 长期续命。
8. 如果实施细则已经把某个旧目录或旧文件定义为“当前阶段结束前必须删除”，那么新实现落地后必须同步删掉旧路径，不能继续通过 preload、delegate、包装层或内部桥接把旧路径当真实实现。
9. 只要新代码仍依赖旧路径提供真实运行时行为，就视为“旧主路径仍在使用”，不得宣称该模块已经完成迁移。

## 3. 目录冻结规则

Phase 0 起，新代码必须按以下职责落点：

1. `scripts/app/`
   - 用例编排
   - 场景协同
   - coordinator / presenter 装配
   - 允许依赖 `domain` 和 `infra`，但不允许吞掉纯业务规则或 UI 节点树声明
2. `scripts/domain/`
   - 纯业务规则
   - 战斗、功法、商店、库存等领域服务
   - 只允许放“去掉场景/运行时依赖后仍可独立成立”的纯规则、纯状态、纯计算模块
3. `scripts/infra/`
   - 单例适配
   - service locator
   - 外部依赖接入
   - 兼容 facade、adapter、运行时装配入口
4. `scenes/ui/`
   - 可视 UI 结构
   - 列表项、卡片、面板、tooltip 等子场景
   - 所有生产可见 UI 的结构性新增都应优先落到这里，而不是继续写脚本拼树

禁止事项：

1. 不允许把新业务继续堆进 `battlefield.gd`、`battlefield_ui.gd`、`battlefield_runtime.gd`。
2. 不允许把新的可编辑 UI 结构继续直接写进脚本。
3. 不允许把应用编排和领域逻辑继续塞进场景脚本。
4. 不允许把“本应拆到 `domain/` 或子模块”的逻辑继续压回旧 facade 文件。
5. 不允许只通过改目录把 runtime / adapter 文件伪装成 `domain`；只要仍依赖场景树、信号、VFX、HexGrid、AutoLoad 或 manager 私有运行时缓存，就不得迁入 `scripts/domain/`。
6. 不允许把本应进入 `scripts/app/` 的用例编排继续写回 `scripts/board/`、`scripts/battle/` 或场景脚本。
7. 不允许把本应进入 `scripts/infra/` 的单例/兼容装配继续散落在业务代码里。
8. 不允许新增生产可见 UI，却不提供对应 `scenes/ui/` 子场景或明确的阶段豁免。

## 4. 基线与豁免

### 4.1 基线文件

基线文件路径：

- `tools/baselines/architecture_guard_baseline.json`

用途：

1. 记录当前仓库已经存在的结构债务。
2. 作为后续“禁止新增或恶化”的比较依据。
3. 只记录文件级计数，不记录行号，避免正常挪行导致误报。

### 4.2 豁免文件

豁免文件路径：

- `tools/baselines/architecture_guard_exceptions.json`

豁免条目字段固定为：

1. `rule`
2. `path`
3. `reason`
4. `expires_phase`

规则：

1. 禁止在源码里写 suppression comment 或内联例外。
2. 所有临时豁免必须集中写入该 JSON。
3. `expires_phase` 必须写明豁免最晚在第几个阶段前移除。
4. 新增 `readability_guard`、`module_split_guard` 规则的豁免，也必须走同一份 exceptions 文件。
5. 对“本轮拆分、后续删除”的旧壳文件申请豁免时，`reason` 必须明确写清替代实现批次和删除意图。
6. 任何已经到达或超过 `expires_phase` 的条目，都必须在进入下一阶段前删除；禁止把过期例外继续带入后续阶段。
7. 不允许仅通过改写 `expires_phase` 来给已过期例外续命；若确有阶段调整，必须先同步更新主方案文档和当前实施细则，再修改例外表。
8. 如果 coverage 或结构测试声称“旧路径已删除/不得回流”，测试实现不得通过排除该目录的扫描方式来规避检查。

## 5. 使用方式

### 5.1 只看结构报告

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\architecture_guard.ps1 -ReportOnly
```

### 5.2 接受当前仓库为新基线

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\architecture_guard.ps1 -WriteBaseline
```

### 5.3 跑完整快速校验

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\check.ps1
```

如仅需跳过架构检查：

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\check.ps1 -SkipArchitecture
```

如仅需跳过泄漏门禁：

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\check.ps1 -SkipLeakGuard
```

### 5.4 可读性与模块拆分门禁

`tools/architecture_guard.ps1` 现除原有基线规则外，还会额外检查改动过的 GDScript 文件：

1. `readability_guard`
   - 中文注释比例
   - 函数级中文注释覆盖率
   - 过长的 `文件说明` / `维护约束` 注释块
   - 单行长度
   - 单行函数体 / 单行控制流
   - 函数体长度
   - 大文件空行密度
2. `module_split_guard`
   - 是否触碰了仍然 `>900` 行的超长文件

这里的“改动文件”优先按 `ARCH_GUARD_BASE_REF...HEAD` 计算；若未提供该环境变量，则按本地工作区相对 `HEAD` 的差异计算。

对“旧目录必须删除”一类规则，还应遵守两条补充约束：

1. 覆盖检查可以排除测试目录，但不得排除当前阶段要求删除或禁止回流的生产目录。
2. 如果测试目标是“旧入口不再存在”，应优先通过文件存在性断言和全仓搜索断言证明，而不是只依赖部分目录拼接文本。

## 6. 泄漏归零门禁

为了把“测试泄漏归零”变成长期约束，`check.ps1` 默认会执行泄漏门禁：

1. 读取 `tools/baselines/leak_guard_tests.json` 的测试清单。
2. 用 `Godot --headless --verbose --script` 逐个执行清单中的测试。
3. 命中以下任一输出即失败：
   - `Leaked instance:`
   - `WARNING: ObjectDB instances leaked at exit`
   - `WARNING: <N> RIDs of type "<Type>" were leaked.`
   - `ERROR: <N> resources still in use at exit`

维护规则：

1. 新增关键回归测试时，应同步评估是否加入 `leak_guard_tests.json`。
2. 不允许通过“忽略 warning”规避泄漏问题；必须修复测试资源释放。
3. 仅在本地调试时允许使用 `-SkipLeakGuard`，合入前必须恢复默认门禁。
4. 如果关键测试缺少最小数据集，必须在 `mods/test/` 下补测试专用 Mod，而不是继续依赖失效目录或脚本内联大段 JSON。

## 7. Review Checklist

提交评审前，至少确认以下事项：

1. 是否新增了 `>600` 行的脚本，或让存量超长脚本继续增长。
2. 是否在 `scripts/main`、`scripts/ui`、`scripts/board` 中新增了 `*.new()` 拼 UI。
3. 是否新增了 `_get_root_node(...)`。
4. 是否在未豁免路径里新增了 `call(...)`。
5. 改动文件是否满足中文注释比例和可读性门禁。
6. 注释是否主要贴着函数、关键参数和复杂分支，而不是堆在文件头部。
7. 是否存在超过限制的 `文件说明` / `维护约束` 注释块。
8. 是否存在单行函数体、单行控制流、超长函数、删除空行凑行数等反模式。
9. 如果触碰了超长文件，是否真的完成了模块迁移，而不是只压缩排版。
10. 如果实施细则要求删除旧路径，当前改动是否真正删除了旧路径，而不是继续隐藏 preload/delegate 依赖。
11. 结构/coverage 检查是否错误地排除了本应被禁止回流的旧目录。
12. 是否仍携带已经到期的 `architecture_guard_exceptions.json` 条目。
13. 新的 UI 是否已经先建 `tscn` 或子场景。
14. 新的应用编排、领域逻辑、基础设施代码是否落到正确目录。
15. 新增或修复测试时，如果依赖额外数据，是否已经优先落到 `mods/test/` 而不是散落在临时路径。

## 8. Phase 0 结束标准

Phase 0 完成后，应满足：

1. 当前结构债务被基线固定。
2. 后续新提交不能再继续污染旧结构。
3. 所有人对“新代码该写到哪里、什么模式禁止再加”没有口径分歧。
4. 不再允许通过压缩代码排版来伪装“完成拆分”。
