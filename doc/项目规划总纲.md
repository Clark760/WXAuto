## 一、项目概述

### 1.1 游戏类型

**武侠自走棋** —— 大规模回合制策略游戏。玩家扮演武林门派掌门人，招募江湖侠客，排兵布阵，通过功法修炼与技能联动，在大规模战场上击败强敌。

**核心差异化卖点**：
- 🏯 **武侠世界观**：江湖门派、功法秘籍、内力修为构成的完整武侠体系
- ⚔️ **大规模战斗**：玩家最多 **50 名角色** 同时上场，敌方规模可超过 **200 人**
- 🔗 **功法联动**：取代传统羁绊系统，通过角色功法与技能的搭配产生联动效果
- 🎨 **纯 2D 画面**：水墨武侠风格 2D 精灵与特效
- 🔧 **全面 Mod 支持**：所有游戏数据（角色、功法、装备、关卡等）均为外部 JSON 配置，玩家可自由定制

### 1.2 核心体验目标

| 维度 | 目标 |
|------|------|
| 沙场气魄 | 百人混战的武侠大场面，刀光剑影、气劲纵横 |
| 策略深度 | 功法流派搭配 + 阵型布局 + 兵力调度，决策层次丰富 |
| 武侠沉浸 | 内力、身法、暗器、阵法等武侠元素深度融入 |
| 成长体验 | 角色修炼进阶、秘籍收集、门派壮大 |
| 重玩性 | 随机秘籍 + 多元流派 + 敌方阵容变化 |

### 1.3 目标平台

- **仅 Windows PC**，单机离线，不含任何联网功能
- 不考虑移动端适配

### 3.2 棋盘与战场系统

由于规模远超传统自走棋，采用**大地图战场**设计：

| 参数 | 规格 |
|------|------|
| 战场格式 | 六边形网格，大地图 |
| 己方部署区 | 约 16×8 = 128 格部署区域 |
| 战场总面积 | 约 32×16 = 512 格可战斗区域 |
| 备战席 | 最多暂存 50 名侠客 |
| 上场上限 | 玩家最多 **50 名**角色同时上场 |
| 敌方规模 | 最多可达 **200+** 敌人（Boss 关卡/攻城战） |

**战场类型**（后期可扩展）：
- 🏔️ 华山之巅 — 标准平地战场
- 🌲 竹林幽径 — 含地形障碍（竹林阻挡视野与移动）
- 🏯 攻城战 — 进攻方攻打城墙，防守方据城而守
- ⛩️ 武林大会 — 擂台赛制，逐轮淘汰

### 3.3 角色系统

#### 角色资源规格

每个角色需要 **两张 PNG 图片**：

| 资源 | 用途 | 路径约定 | 说明 |
|------|------|---------|------|
| **战斗精灵图** | 战场上的角色显示 | `assets/sprites/units/{unit_id}.png` | 小尺寸，用于 MultiMesh 批量渲染 |
| **立绘图** | 角色详情面板、招募商店等 UI | `assets/sprites/portraits/{unit_id}.png` | 大尺寸半身/全身像 |

#### 角色视觉表现（程序化精灵动画）

角色**不使用序列帧动画**。战斗精灵图为单张静态 PNG，通过 `Tween` 程序化动画模拟动作。

以下为 `sprite_animator.gd` 需实现的动画状态及其 Tween 参数：

| 状态枚举值 | Tween 变换 | 默认参数 |
|-----------|-----------|----------|
| `IDLE` | `position.y` 正弦波浮动 | 幅度 ±2px，周期 1.2s |
| `MOVE` | `rotation` 左右交替 + `position` 水平位移 | 倾斜 ±5°，频率 0.3s |
| `ATTACK` | `position` 朝目标方向前冲 → 回弹 | 前冲 5px，时长 0.15s |
| `SKILL` | `scale.y` 拉伸 + `modulate` 发光闪烁 | 拉伸 1.2x，闪 2 次 |
| `HIT` | `position` 后退 + `modulate` 红闪 | 后退 4px，红闪 0.1s |
| `DEATH` | `scale` 缩小 + `modulate.a` → 0 + `position.y` 下沉 | 时长 0.5s |
| `VICTORY` | `position.y` 连续跳动 + `scale` 脉冲 | 跳高 8px，脉冲 1.1x |
| `BENCH` | `rotation` 缓慢摇晃 | ±3°，周期 2s |

所有动画参数均可通过角色 JSON 中的 `animation_overrides` 字段覆盖。
Mod 制作者添加新角色仅需提供 **一张战斗精灵图 PNG + 一张立绘 PNG**，无需制作动画帧。

### 3.4 功法系统（核心差异化系统）

> 取代传统自走棋的羁绊系统，功法系统是本游戏的**核心创新机制**。

#### 3.4.1 功法分类

| 功法类型 | 说明 | 示例 |
|----------|------|------|
| **内功心法** | 被动增益，影响基础属性与内力恢复 | 九阳神功、紫霞神功、蛤蟆功 |
| **外功招式** | 主动技能，攻击/防御/控制 | 降龙十八掌、独孤九剑、六脉神剑 |
| **身法** | 移动与位移技能 | 凌波微步、梯云纵、壁虎游墙 |
| **阵法** | 部署型，对区域内友方施加增益 | 天罡北斗阵、真武七截阵、金刚伏魔阵 |
| **奇术** | 特殊效果，如复活、变身、召唤 | 乾坤大挪移、化功大法、七伤拳 |

#### 3.4.2 功法联动机制

**功法联动**是角色间通过功法搭配产生的组合增益效果，取代传统羁绊系统：

**联动类型一：同门联动**
- 同一门派的角色同时上场且修炼相关功法时触发
- 例：3 名修炼「全真剑法」的武当弟子站位相邻 → 自动组成「三才剑阵」，攻击力 +30%、互相格挡 +15%

**联动类型二：功法共鸣**
- 不同角色修炼的功法之间存在五行相生关系
- 例：一名角色修炼「纯阳功」（火属性），邻近角色修炼「归元功」（木属性）→ 木生火共鸣，「纯阳功」伤害 +25%

**联动类型三：武技连携**
- 特定技能搭配产生额外效果
- 例：角色 A 释放「掌力」击飞敌人，角色 B 在落点使用「剑气」→ 触发「掌剑连击」，额外造成 200% 伤害

**联动类型四：阵法联动**
- 阵法类功法覆盖的区域内，特定功法的角色获得增强
- 例：「天罡北斗阵」范围内修炼「全真内功」的角色，攻防 +40%，且受伤时阵内友方分摊伤害

**联动发现机制**：
- 部分联动明示（功法描述中提及），部分需要玩家探索发现
- 游戏内「功法录」逐步解锁联动图谱，鼓励尝试

#### 3.4.3 功法获取

- **招募自带**：角色自带 1-2 个初始功法
- **秘籍掉落**：战斗胜利后随机获得功法秘籍
- **秘籍商店**：花银两购买
- **修炼领悟**：角色升星时概率领悟新功法

### 3.7 战斗逻辑

#### 行为决策树

```
角色每帧行为决策：
┌─ 是否有阵法效果需要维持？
│  └─ 是 → 维持阵法站位
├─ 是否有攻击目标在范围内？
│  ├─ 有 → 检查内力是否充足
│  │   ├─ 内力足 → 释放功法技能（优先级排序）
│  │   │   └─ 检查是否触发联动效果
│  │   └─ 内力不足 → 普通攻击（回复内力）
│  └─ 无 → 使用身法/移动寻找最近目标
├─ 是否为医者类角色？
│  └─ 检查友方气血 → 优先治疗低血量友方
└─ 统领类角色
   └─ 维持光环范围，跟随主力群移动
```

#### 伤害公式

```
外功伤害 = ATK × 技能倍率 × (100 / (100 + 目标DEF)) × 随机浮动(0.9~1.1)
内功伤害 = IAT × 技能倍率 × (100 / (100 + 目标IDR)) × 随机浮动(0.9~1.1)
闪避率 = (自身SPD - 对方SPD) / (自身SPD + 100)  [下限0, 上限40%]
暴击率 = 基础5% + 功法加成
暴击伤害 = 150% + 功法加成
```

#### 战斗特效方案（纯程序化，零外部美术资源）

所有战斗特效**不使用任何外部特效贴图/精灵帧**，完全通过 Godot 内置节点与 Shader 实现：

| 特效类型 | 实现方式 | Godot 节点/API | 示例 |
|----------|---------|---------------|------|
| **剑气/掌风** | 拖尾粒子 + 方向拉伸 | `GPUParticles2D` + `ParticleProcessMaterial` | 白色→淡蓝渐变粒子扇形扩散 |
| **内力光环** | 圆形发光扩散 | `GPUParticles2D`（环形发射） | 金色粒子环绕角色旋转 |
| **伤害数字** | 浮动文字 | `Label` + `Tween`（上浮+淡出） | "-128" 红色上浮消失 |
| **治疗效果** | 上升粒子 | `GPUParticles2D`（重力反向） | 绿色光点上升 |
| **暗器/飞弹** | 直线运动投射物 | `Line2D` 拖尾 + `Sprite2D` 简单形状 | 从A到B的飞行线 |
| **阵法光圈** | 地面圆环 | `_draw()` 自定义绘制 / `Line2D` 圆形 | 半透明旋转法阵圆环 |
| **范围指示器** | 地面高亮 | `_draw()` + Shader 半透明 | 技能范围预览 |
| **受击闪白** | 全身闪色 | `ShaderMaterial`（modulate 覆盖） | 受击瞬间白闪 0.05s |
| **毒/灼烧DOT** | 持续粒子附着 | `GPUParticles2D`（跟随角色） | 紫色/橙色小粒子持续冒出 |
| **击飞轨迹** | 抛物线拖尾 | `Line2D`（点动态追加）+ `Tween` | 弧形位移路径 |

**`vfx_factory.gd`** 负责管理所有特效的创建与对象池回收。特效参数完全数据驱动。

**特效 JSON 配置示例**（`data/vfx/sword_qi.json`）：
```json
{
  "id": "vfx_sword_qi",
  "type": "particles",
  "emitter": "cone",
  "amount": 24,
  "lifetime": 0.4,
  "color_start": "#FFFFFF",
  "color_end": "#88CCFF00",
  "speed": 200,
  "spread_angle": 30,
  "scale_curve": [1.0, 0.3],
  "gravity": {"x": 0, "y": 0}
}
```

Mod 制作者可通过新增/覆盖 `data/vfx/*.json` 自定义特效参数（颜色、粒子数、速度、生命周期等），也可替换 `assets/shaders/` 下的 Shader 文件。

#### 大规模战斗优化要点

由于战场上可能存在 **250+ 个角色**同时活动，需要特别的性能设计：

- **分组 AI**：将角色按阵营/区域分组，组内共享寻敌结果
- **LOD 行为**：屏幕外/远距离角色简化 AI 更新频率
- **空间分区**：使用四叉树/网格分区加速碰撞检测与寻敌
- **逻辑帧分离**：战斗逻辑以固定帧率（如 10fps 逻辑帧）运行，渲染帧插值
- **对象池**：所有角色、程序化特效、伤害数字复用对象池
- **批量渲染**：使用 MultiMeshInstance2D 批量渲染同类型角色

### 4.3 关键设计模式

| 模式 | 应用场景 | 说明 |
|------|---------|------|
| **单例** | GameManager, EventBus, ObjectPool | Godot AutoLoad |
| **观察者** | EventBus | 系统间解耦（信号） |
| **状态机** | 游戏阶段、角色行为 | 清晰管理状态转换 |
| **工厂** | UnitFactory | 从 Resource 数据创建实例 |
| **策略** | 功法效果、寻敌策略 | 可插拔行为 |
| **组件** | 角色（战斗/移动/动画拆分） | 组合优于继承 |
| **享元** | MultiMesh 批量渲染 | 大量同类对象共享数据 |
| **命令** | 战斗结算控制 | 记录关键状态并触发即时停战 |
| **空间哈希** | 碰撞/寻敌 | 加速大规模空间查询 |

### 4.4 大规模战斗性能方案

这是项目最大的技术难点，**250+ 角色同屏**要求：

| 层面 | 方案 | 细节 |
|------|------|------|
| **渲染** | MultiMeshInstance2D | 同类角色合批渲染，大幅减少 Draw Call |
| **逻辑帧** | 固定 10fps 逻辑帧 | 战斗逻辑低频更新，渲染帧插值平滑 |
| **空间查询** | SpatialHash 网格分区 | O(1) 邻近查询替代 O(n²) 遍历 |
| **AI 分组** | Group AI | 相近角色共享寻敌，减少决策计算 |
| **LOD 行为** | 远距离/屏外简化 | 非可视区域角色降低更新频率 |
| **对象池** | ObjectPool | 预创建 300+ 角色实例循环利用 |
| **特效池** | 程序化特效对象池 | GPUParticles2D / Line2D 实例复用 |
| **寻路** | 流场寻路（Flow Field） | 大量角色高效共享寻路结果 |

### 4.5 数据驱动设计（JSON + Mod 可覆盖）

所有游戏数据使用 **JSON 文件** 存储（而非 Godot Resource），以便玩家通过 Mod 自由修改：

| JSON 文件 | 字段 |
|-----------|------|
| **units/*.json** | ID、名称、势力归属、品质、费用、基础属性、定位、初始功法列表、战斗精灵图路径、立绘图路径、animation_overrides（可选） |
| **gongfa/*.json** | ID、名称、类型（内功/外功/身法/阵法/奇术）、五行属性、效果ID、冷却、内力消耗、技能倍率、联动标签 |
| **linkages/*.json** | ID、名称、触发条件（功法组合/站位/五行关系）、效果描述、增益数值 |
| **equipment/*.json** | ID、名称、类型、属性加成、合成配方、被动效果 |
| **enemies/*.json** | 关卡ID、敌方阵容列表、AI 策略、特殊规则 |
| **levels/*.json** | 等级、所需经验、上场上限、商店概率表 |

每个数据目录下附带 `_schema/` 文件夹，提供 JSON Schema 供 Mod 制作者校验数据格式。

### 4.6 Mod 系统架构

> Mod 支持是本项目的**设计级需求**，所有系统均围绕可 Mod 化设计。

#### Mod 加载流程

```
游戏启动
  → 加载 res://data/ 下的基础数据（JSON）
  → 扫描 user://mods/ 目录下所有 Mod 文件夹
  → 按 mod.json 中的 load_order 排序
  → 依次加载各 Mod 的 data/ 和 assets/
  → 同 ID 数据：后加载的 Mod 覆盖先加载的
  → 新 ID 数据：追加到数据池
  → 完成数据合并，进入主菜单
```

#### Mod 文件结构

```json
// mods/my_mod/mod.json
{
  "id": "my_wuxia_mod",
  "name": "我的武侠扩展包",
  "author": "玩家A",
  "version": "1.0.0",
  "description": "新增昆仑派与 10 个角色",
  "game_version_min": "1.0.0",
  "load_order": 100,
  "overwrites": ["units/kunlun_*.json"],
  "additions": ["gongfa/kunlun_*.json", "linkages/kunlun_*.json"]
}
```

#### Mod 可定制内容

| 内容 | 方式 | 说明 |
|------|------|------|
| **角色** | 新增/修改 JSON + 精灵图 + 立绘 | 自定义属性、定位、功法列表 |
| **功法** | 新增/修改 JSON | 自定义技能数值、联动标签 |
| **联动关系** | 新增/修改 JSON | 自定义触发条件与效果 |
| **装备** | 新增/修改 JSON | 自定义属性、合成配方 |
| **敌方阵容** | 新增/修改 JSON | 自定义关卡敌人配置 |
| **关卡配置** | 新增/修改 JSON | 自定义等级表、概率表 |
| **势力/门派** | 新增 JSON | 添加全新门派体系 |
| **战斗特效** | 新增/修改 `data/vfx/*.json` | 自定义粒子颜色、数量、速度等参数 |
| **Shader** | 替换 `assets/shaders/` 文件 | 自定义特效着色器 |
| **精灵图/立绘** | 替换图片文件 | 角色战斗图、立绘、UI 替换 |
| **音效/音乐** | 替换音频文件 | 自定义 BGM 与音效 |
| **数值公式** | 修改配置 JSON | 调整伤害、概率等数值参数 |

#### Mod 工具支持

- 游戏内 **Mod 管理器** UI：启用/禁用/排序已安装 Mod
- 提供 **JSON Schema** 文件，Mod 制作者可用 VS Code 等编辑器获得自动补全与校验
- 提供 **示例 Mod**（`mods/example_mod/`）作为模板
- 数据加载时自动校验 JSON 格式，错误数据跳过并记录日志

## 七、技术要点与注意事项

### 7.1 Godot 4.6 特性利用

| 特性 | 应用 |
|------|------|
| **GDScript 2.0 静态类型** | 全面使用类型标注，提升性能与可维护性 |
| **Resource** | 数据驱动设计核心，所有游戏数据资源化 |
| **Signal** | 系统解耦通信 |
| **MultiMeshInstance2D** | 大规模角色批量渲染 |
| **Tween** | UI 动效与战斗表现 |
| **NavigationServer** | 辅助大规模寻路 |
| **RenderingServer** | 底层渲染控制优化 |
| **Theme + StyleBox** | 统一武侠 UI 风格 |

### 7.2 大规模战斗性能指标

| 指标 | 目标值 |
|------|--------|
| 同屏角色数 | 250+ |
| 逻辑帧率 | 10 fps |
| 渲染帧率 | 60 fps |
| 单帧逻辑耗时 | < 8ms |
| 内存占用 | < 1.5 GB |
| Draw Call | < 100（MultiMesh 合批） |

### 7.3 可扩展设计（面向 Mod）

- **数据与逻辑分离**：所有游戏数据为外部 JSON，代码仅读取数据
- 功法效果采用策略模式，预置效果类型覆盖常见需求
- 联动关系纯数据驱动，新增联动无需改代码
- 角色全部数据驱动，Mod 新增角色零代码
- Mod 加载器支持覆盖与追加两种模式

## 八、风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| **250+ 角色性能** | 卡顿、体验差 | 逻辑帧分离 + MultiMesh + 空间哈希 + 对象池 |
| 功法联动复杂度膨胀 | 难以平衡 | 分阶段设计联动，先实现核心后扩展 |
| 大规模战斗可读性差 | 看不懂打了什么 | 镜头聚焦 + 伤害统计 + 关键角色高亮 |
| 角色数量多美术工作量大 | 开发周期长 | 每角色仅2张PNG（精灵图+立绘），无序列帧，可AI生成 |
| 功法数值平衡困难 | 某流派过强 | 建立数值框架 + 持续迭代 + 数据驱动调参 |
| AI 在大规模下的合理性 | AI 行为愚蠢 | 分组 AI + 预设阵型模板 |
