# M5新版联动效果（Effect 化 Tag 联动）

## 1. 目标

本方案用于替代旧联动机制，统一通过 `effect` 执行“按 Tag 检测并触发效果”。

核心需求：
- 支持在半径 `n` 格内检索 `unit 特性 / 功法 / 装备 / 地形 / unit 本体` 的 Tag。
- `n = 0` 时只检查：
  - 自身特性 Tag
  - 自身功法 Tag
  - 自身装备 Tag
  - 自身 unit Tag
  - 当前所在格地形 Tag
- 支持多 Tag 条件（`any/all`）。
- 支持按匹配数量分段触发不同效果（如 3/5/7 档）。
- 不保留旧联动兼容链路，统一以 `tags` 字段为唯一来源。

---

## 2. 统一标签规范

### 2.1 数据字段
- `unit.tags`：单位本体标签。
- `unit.traits[].tags`：特性标签。
- `gongfa.tags`：功法标签。
- `equipment.tags`：装备标签。
- `terrain.tags`：地形标签（定义表 + 运行时覆盖）。

### 2.2 规范化规则
- 统一 `trim + to_lower + 去重`。
- 标签命名建议使用命名空间，降低冲突：
  - `array.zhenwu`
  - `element.poison`
  - `element.fire`
  - `trait.wandu_guxin`

### 2.3 计数口径
- 默认 `count_mode = provider`：同一个提供者（同一 trait/gongfa/equipment/terrain 实例）只计一次。
- 可选 `count_mode = occurrence`：每次命中都计数（一般不推荐，易膨胀）。
- 可选 `count_mode = unit`：同一单位最多记 1 次（适合“附近有多少单位满足”）。

---

## 3. 新增 Effect：`tag_linkage_branch`

### 3.1 作用
在执行时扫描附近 Tag，按条件分支执行不同 `effects`。

### 3.2 建议结构

```json
{
  "op": "tag_linkage_branch",
  "range": 5,
  "include_self": true,
  "team_scope": "ally",
  "source_types": ["trait", "gongfa", "equipment", "terrain", "unit"],
  "count_mode": "provider",
  "queries": [
    {
      "id": "q_array",
      "tags": ["array.zhenwu"],
      "tag_match": "any",
      "source_types": ["trait", "gongfa", "equipment", "unit"]
    }
  ],
  "cases": [
    {
      "id": "tier_7",
      "all": [{ "query_id": "q_array", "min_count": 7 }],
      "effects": []
    },
    {
      "id": "tier_5",
      "all": [{ "query_id": "q_array", "min_count": 5 }],
      "effects": []
    },
    {
      "id": "tier_3",
      "all": [{ "query_id": "q_array", "min_count": 3 }],
      "effects": []
    }
  ],
  "else_effects": [],
  "stop_after_first_case": true
}
```

### 3.3 字段定义
- `range`: 搜索半径（六边形距离）。
- `include_self`: 是否包含自身。
- `source_types`: 全局来源过滤：`trait / gongfa / equipment / terrain / unit`。
- `queries`: 查询定义。
  - `id`: 查询标识。
  - `query_type`: `match_tags`（默认）或 `forbid_tags`。
  - `tags`: 要匹配的标签列表。
  - `tag_match`: `any` 或 `all`。
  - `exclude_tags`（可选）:
    - `query_type=match_tags` 时，作为“provider 级排除条件”（先命中 `tags`，再剔除命中 `exclude_tags` 的 provider）。
    - `query_type=forbid_tags` 时，作为“违规标签集合”（命中即计数）。
  - `exclude_match`（可选）: `any` 或 `all`（默认 `any`），用于 `exclude_tags` 的命中策略。
  - `source_types`（可选）: 覆盖全局来源过滤。
  - `origin_scope`（可选）: `self / nearby / all`。
- `count_mode`: `provider / occurrence / unit`。
- `cases`: 条件分支（按顺序判定）。
  - `all`: 全部满足才命中。
  - `any`（可选）: 任意满足即命中。
  - 条件项：`query_id + min_count (+ max_count)`。
- `stop_after_first_case`: 命中首分支后是否停止。
- `else_effects`: 无分支命中时执行。

### 3.4 `n = 0` 语义
当 `range = 0`：
- 不扫描附近单位。
- 仅采集自身来源（trait/gongfa/equipment/unit）。
- 仅采集当前 cell 的 terrain tags。

### 3.5 负向判定示例

#### 示例 A：X 格内不存在火属性 Tag

```json
{
  "id": "q_no_fire",
  "query_type": "forbid_tags",
  "exclude_tags": ["element.fire"],
  "exclude_match": "any",
  "source_types": ["terrain"]
}
```

配合 case 条件：

```json
{ "query_id": "q_no_fire", "max_count": 0 }
```

语义：范围内没有任何 provider 命中 `element.fire`。

#### 示例 B：必须有剑阵标签，但不能带眩晕标签

```json
{
  "id": "q_clean_array",
  "query_type": "match_tags",
  "tags": ["array.zhenwu"],
  "tag_match": "any",
  "exclude_tags": ["debuff.stun"],
  "exclude_match": "any",
  "source_types": ["unit", "trait"]
}
```

语义：只统计命中 `array.zhenwu` 且未命中 `debuff.stun` 的 provider。

## 4. 判定流程（实现约定）

1. 取施法者当前格 `origin_cell`。
2. 按 `range` 计算扫描格集合：
   - `range = 0` -> 仅 `origin_cell`。
   - `range > 0` -> BFS 半径格集合。
3. 收集 Tag 提供者（Provider）：
   - Unit 来源：筛选存活单位、team_scope、距离。
   - Terrain 来源：扫描格逐格读取 `CombatManager.get_terrain_tags_at_cell(cell)`。
4. 对每个 `query` 统计命中数。
5. 依次判定 `cases`，执行首个命中分支（若 `stop_after_first_case=true`）。
6. 若均未命中，执行 `else_effects`。
7. 分支 `effects` 复用现有执行链（再次 `execute_active_effects`），不走旧联动接口。

### 4.1 触发与重算策略（事件驱动优先）

不建议把联动长期依赖高频轮询（`passive_aura + 短 Buff 刷新`）。  
推荐改为“状态驱动 + 脏标记重算”：

1. 施法者移动（离开旧格/进入新格）。
2. 周围目标移动。
3. 周围单位死亡、召唤、入场、退场。
4. 单位标签源变化（特性、功法、装备发生变更）。
5. 地形变化（进入/离开/覆盖/过期）。

实现建议：
- 给带 `tag_linkage_branch` 的单位建立“监听引用”（按 `range` 订阅周边 cell / 单位）。
- 事件触发时仅标记 `dirty`，由调度器在后续物理帧重算。
- 结算采用“长效 Buff 增删”而不是无限刷新短效 Buff，避免抖动与日志噪声。

```json
              "id": "tier_7",

            },
            {
              "id": "tier_5",
              "all": [{ "query_id": "q_array", "min_count": 5 }],
```

## 8. 代码编写指南（按文件）

### 8.1 新增解析器
文件：`scripts/gongfa/tag_linkage_resolver.gd`

建议接口：

```gdscript
class_name TagLinkageResolver
```

### 8.5 Tag 索引与位掩码（性能基线）

为避免“扫描 + 字符串匹配”在大规模单位下成为瓶颈，建议在启动/加载期构建全局 Tag 索引：

- `TagRegistry`：
  - `tag_to_index: Dictionary<String, int>`
  - `index_to_tag: Array<String>`
- 每个 Provider 预计算：
  - `tag_indices: PackedInt32Array`（稀疏）
  - `tag_mask: PackedInt64Array`（按 64bit 分片，适合 `all/any` 位运算）

匹配规则建议：
- `tag_match=any`：`(provider_mask & query_any_mask) != 0`
- `tag_match=all`：`(provider_mask & query_all_mask) == query_all_mask`
- 超过 64 个 tag 时使用分片数组逐片位与。

若不使用 BitMask，至少使用整数 Hash（如稳定 FNV-1a）并在查询前把字符串转为整型集合，避免运行时反复 `to_lower + 字符串比较`。

### 8.6 事件驱动评估器（替代高频轮询）

新增运行时模块建议：`TagLinkageRuntimeScheduler`

职责：
- 维护 `owner_id -> linkage_watcher`。
- 建立 `cell -> watchers`、`unit_id -> watchers` 的反向引用。
- 事件到达时只打脏标记：`watcher.dirty = true`。
- 在调度阶段执行 `evaluate()` 并差量增删 Buff。

推荐接入事件：
- `unit_moved(unit, from_cell, to_cell)`
- `unit_spawned(unit)` / `unit_died(unit)`
- `unit_tags_changed(unit)`（特性/功法/装备变更）
- `terrain_changed(cell)`（创建、覆盖、过期）

### 8.7 `effect_engine.gd` Tick 错峰执行

即使事件驱动，也需要兜底重扫（防漏事件或跨系统状态漂移），但必须错帧：

```gdscript
func _should_run_linkage_scan_this_frame(owner: Node, buckets: int, physics_frame: int) -> bool:
	var bucket_count: int = maxi(buckets, 1)
	var phase: int = posmod(owner.get_instance_id(), bucket_count)
	return posmod(physics_frame, bucket_count) == phase
```

调度建议：
- 在物理帧里，只有命中自己相位的单位才允许执行兜底扫描。
- `buckets` 可按战场单位数动态调整（如 4/8/16 档）。
- 事件触发的 `dirty` 单位优先执行，不受错峰限制；非脏单位才走错峰兜底。

## 10. 性能与稳定性建议

- 优先事件驱动，轮询只作兜底。
- 使用全局 Tag 索引（BitMask 或整数 Hash），避免高频字符串比较。
- 维护 Provider 缓存与脏标记，避免全量重建 Provider 列表。
- 增量更新 Buff（增删状态）替代“无限刷新短效 Buff”。
- 兜底扫描必须错峰：按 `unit.instance_id % bucket_count` 分帧执行。
- 调试期输出 `query_counts/case_id/watcher_id` 到战斗日志，便于验收。
