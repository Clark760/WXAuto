## 问题一：后排单位在友方身后两格间来回震荡

### 精确根因

**根因链**：

```
block_teammate_cells_in_flow = false（combat_manager.gd 行 32）
  → FlowField.build() 时友方占格被视为可通行，分配了 cost 值
  → _pick_best_adjacent_cell() 中 _is_cell_free(neighbor) 过滤掉友方占格
  → 所有实际可走的空邻格 flow cost ≥ 当前格 cost
  → 没有下坡格（downhill）可选
  → allow_equal_cost_side_step = true 且 cost > 2 → 选同成本空邻格横移
  → 下一帧流场方向不同 → 横移回来
  → 震荡
```

**具体场景**：

```
  后排单位 A 在格 (3,2)，flow cost = 4
  友方 B 在格 (4,2)，flow cost = 3（流场认为可通行但实际被占）
  友方 C 在格 (3,1)，flow cost = 3（同上）
  空格 (2,2) flow cost = 5（远离敌人）
  空格 (4,3) flow cost = 4（同成本）
  空格 (2,1) flow cost = 4（同成本）

  A 没有 cost < 4 的可走空格 → allow_equal_cost_side_step 选 (4,3) → 移动
  下一帧 A 在 (4,3)，flow cost = 4，(3,2) 也是 cost 4 → 横移回来
```

### 修改方案

**修改 1（核心 — 启用友方阻挡流场）**：

```
文件: combat_manager.gd
行号: 32
```

```gdscript
# 修改前:
@export var block_teammate_cells_in_flow: bool = false

# 修改后:
@export var block_teammate_cells_in_flow: bool = true
```

**说明**：将 `block_teammate_cells_in_flow` 改为 `true`，使流场构建时把友方占格标记为 blocked。这样 BFS 会**绕开**友方占格计算 cost，后排单位的流场会自然引导它们绕过友方前排。

> [!WARNING]
> 之前注释说"高密度战斗默认采用软阻挡…避免大面积 -1 无路"（行 611-613）。启用后可能在极拥堵场景导致部分格子不可达（cost=-1）。需配合修改 2 兜底。

**修改 2（兜底 — 不可达时禁用横移和上坡）**：

当 `flow_cost < 0`（不可达）时，当前代码仍会尝试横移/上坡移动。应增加守卫：

```
文件: combat_manager.gd
函数: _pick_best_adjacent_cell
行号: 在 1178-1181 之后（current_cost 赋值后）增加
```

```gdscript
    if current_cost < 0:
        best_cost = INF
    else:
        best_cost = float(current_cost)

    # 新增：当前格不可达且所有邻格都被占时，原地不动
    # 避免在不可达区域仍然被横移/上坡驱动震荡
    var any_free_neighbor: bool = false
    for check_neighbor in _neighbors_of(current_cell):
        if _is_cell_free(check_neighbor):
            any_free_neighbor = true
            break
    if not any_free_neighbor:
        return current_cell
```

**修改 3（可选优化 — 彻底禁用上坡逃逸步）**：

```
文件: combat_manager.gd
行号: 34
```

```gdscript
# 修改前:
@export var allow_uphill_escape_step: bool = true

# 修改后:
@export var allow_uphill_escape_step: bool = false
```

**说明**：上坡步的设计初衷是防止单位被友方完全堵死，但实际效果是导致后排单位远离目标再靠近的震荡。在 `block_teammate_cells_in_flow=true` 后，流场已经正确引导绕路，不再需要上坡逃逸作为兜底。

## 问题二：点击"继续"后 VICTORY 动画不回 IDLE

### 精确根因

`battlefield.gd` 中已实现 `reset_all_units_to_idle()`（行 255-265），但**从未被任何代码调用**。

调用链分析：
```
玩家点击"继续" → battlefield._set_stage(Stage.PREPARATION)
  → super._set_stage(next_stage)  // 父类 battlefield_ui.gd
  → _apply_stage_ui_state()
    → battle_flow.sync_stage(_stage, Stage.RESULT)
      → 只调 hide_stats_panel()  ← 没有调 reset_all_units_to_idle()
```

### 修改方案

**修改点：`_apply_stage_ui_state()` 中调用 `reset_all_units_to_idle()`**

```
文件: battlefield.gd
函数: _apply_stage_ui_state
行号: 180-197，在现有代码的 sync_stage 调用之后增加
```

```gdscript
func _apply_stage_ui_state() -> void:
    super._apply_stage_ui_state()
    if _shop_panel != null and is_instance_valid(_shop_panel):
        if _stage == Stage.PREPARATION:
            _set_shop_panel_visible(_shop_open_in_preparation)
        else:
            _set_shop_panel_visible(false)
    if _recycle_drop_zone != null and is_instance_valid(_recycle_drop_zone):
        var recycle_visible: bool = _stage == Stage.PREPARATION
        _recycle_drop_zone.visible = recycle_visible
        if _recycle_drop_zone.has_method("set_drop_enabled"):
            _recycle_drop_zone.call("set_drop_enabled", recycle_visible)
        if not recycle_visible and _recycle_drop_zone.has_method("clear_external_preview"):
            _recycle_drop_zone.call("clear_external_preview")
    if _battle_flow != null and is_instance_valid(_battle_flow):
        _battle_flow.call("sync_stage", _stage, Stage.RESULT)

    # ========== 新增：离开 RESULT 阶段时，将所有棋子切回 IDLE ==========
    if _stage != Stage.RESULT:
        reset_all_units_to_idle()
    # ==================================================================

    _sync_linkage_panel_visibility()
    _update_shop_ui()
```

**说明**：`_apply_stage_ui_state()` 在**每次阶段切换**时被调用。增加条件 `_stage != Stage.RESULT` 后，当从 RESULT 切到 PREPARATION 时会执行 `reset_all_units_to_idle()`。该方法已存在（行 255-265），会遍历所有部署单位调用 `reset_visual_transform()` + `play_anim_state(0)`。

> [!IMPORTANT]
> 如果 `_stage` 永远不等于 `RESULT`（即 `_set_stage` 从不被设为 RESULT），则此守卫可能不够。备选方案：在 `_set_stage()` 中检测 `previous_stage == Stage.RESULT`：

```gdscript
# battlefield.gd — _set_stage() 修改（行 171-177）：
func _set_stage(next_stage: int) -> void:
    var previous_stage: int = _stage
    super._set_stage(next_stage)

    # 新增：从结算阶段离开时，结束 VICTORY 动画
    if previous_stage == Stage.RESULT and next_stage != Stage.RESULT:
        reset_all_units_to_idle()

    if next_stage == Stage.PREPARATION and previous_stage != Stage.PREPARATION:
        _shop_open_in_preparation = true
        _refresh_shop_for_preparation(false)
    _update_shop_ui()
```

**推荐使用 `_set_stage()` 方案**，因为它精确匹配"从 RESULT 离开"的语义。
