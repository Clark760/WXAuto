# M5 地形统一编码与配置化说明

> **目标读者**：AI 代码生成器 / 核心系统开发者
> **涉及文件**：`terrain_manager.gd`、`hex_grid.gd`、`stage_bridge.gd`、`combat_manager.gd`
> **核心目标**：消除 `terrain_manager.gd` 中的硬编码，将“临时地形”与“关卡固定地形”统一为一套基于 JSON 的配置化模块，支持 Mod 扩展。

---

## 一、当前现状与痛点

目前游戏内的地形（Terrain）分为两类，缺乏统一：
1. **关卡静态地形/障碍 (Stage Static)**：
   - 障碍直接通过 `HexGrid.set_obstacle_cells` 渲染，并在 `CombatManager._static_blocked_cells` 注册，由寻路系统绕过。没有统一的地形属性。
2. **效果临时地形 (Temporary Effects)**：
   - 由功法/装备的 `create_terrain` Op 生成，由 `TerrainManager` 管理。
   - **痛点（硬编码灾难）**：`terrain_manager.gd` 内部存在大量按 `terrain_type` (如 `"fire"`, `"ice"`, `"poison"`) 硬编码的开关：
     - `TERRAIN_COLOR_BY_TYPE`
     - `DEFAULT_DPS_BY_TYPE`
     - `_apply_terrain_tick` 中的巨型 `match` 分支
     - `_default_target_mode_for_terrain`、`_default_debuff_for_terrain` 等

---

## 二、配置化重构方案

### 2.1 新增地形数据表 (`data/terrains/`)

设计一套全新的 JSON 数据结构，代替硬编码常量。不在每个地形独立建立文件，而是在统一的 JSON 文件中以数组的形式存放各类地形的数据（支持分为 core / expansion 多张表让 DataManager 聚合加载）。

**目录结构示例**：
```
data/terrains/
  terrains_core.json
  terrains_m5_expansion.json (支持 Mod 通过增加文件来扩展特定地形包)
```

**JSON 数据格式示例 (`terrains_core.json`)**：
```json
[
    {
        "id": "terrain_fire",
        "name": "烈火",
        "type": "hazard",            // hazard (持续伤害/减益), beneficial (光环/治疗), obstacle (障碍)
        "color": "#F5703357",        // 十六进制颜色 (RGBA)
        "is_barrier": false,         // 是否阻挡寻路机制
        "tick_interval": 0.5,        // 多久结算一次
        "target_mode": "enemies",    // 结算目标：enemies / allies / all
        "base_dps": 25.0,            // 基础秒伤（可被动态 Op 传入的数值覆盖）
        "base_hps": 0.0,             // 基础秒疗
        "damage_type": "internal",   // 伤害物理类型
        "apply_buff_id": "debuff_burn",  // 每次 tick 尝试挂的特定状态（如果不靠单纯扣血）
        "buff_duration": 1.5,
        "vfx_on_tick": "vfx_fire_tick" // 可选，触发时的特效
    },
    {
        "id": "terrain_rock",
        "name": "巨石",
        "type": "obstacle",
        "color": "#80808099",
        "is_barrier": true,
        "target_mode": "none"
    }
]
```

### 2.2 数据加载机制注入

1. **DataManager & Registry**：
   - 扩展数据加载模块，允许读取 `data/terrains/` 下所有 JSON，装载进内存。
2. **TerrainManager 初始化**：
   - 给 `TerrainManager` 添加一个属性引用 `terrain_registry`，读取全局地形表。

## 三、代码更新说明 (修改点 Checklist)

### 1. `terrain_manager.gd` 改造

- **删除常量**：彻底删除 `TERRAIN_COLOR_BY_TYPE`、`DEFAULT_DPS_BY_TYPE`及后缀为 `_for_terrain` 的各种匹配方法。
- **构建解析层**：
  - 更新 `_build_terrain_entry`：从配置表检索 `terrain_type` (如传进来 "fire" 就去找 "terrain_fire")。
  - 读取字典合并默认属性。例如 `config.get("damage_per_second", json_cfg.base_dps)`。
- **重构 Tick 循环 (`_apply_terrain_tick`)**：
  - **移除巨型 `match` 分支**。
  - 改为数据驱动：
    ```gdscript
    # 务必保留在临时地形中的 source_fallback 以兼容 M5 战斗数据统计
    var source_fallback: Dictionary = _build_source_fallback_from_terrain(terrain)
    
    if damage_per_tick > 0.0:
        combat.apply_environment_damage(unit, damage_per_tick, source, json_cfg.damage_type, source_fallback)
    if heal_per_tick > 0.0:
        combat.restore_hp(heal_per_tick) # 注：由于目前治疗统计强依赖 combat，若需全量统计地形治疗，此处也应跟进抛出信号。
    if not json_cfg.apply_buff_id.is_empty():
        gongfa.apply_runtime_buff(unit, json_cfg.apply_buff_id, json_cfg.buff_duration, source, "terrain")
    ```
- **融合静态地形（Static Terrains）**：
  - 加一个 `add_static_terrain(terrain_type, cells_array)` 接口。
  - 生成 `remaining = -1.0` (永久) 的底层对象。`is_barrier` 根据配置自动注入阻挡格。

### 1.5 伤害统计模块 (`BattleStatistics`) 兼容保障

在进行解耦与读表化改造时，**必须保留地形伤害归属追溯（`source_fallback`）机制**：
- 目前临时地形（由单位技能生成）通过 `_build_source_fallback_from_terrain` 获取 `source_id`, `source_name`、`source_team`。当施法者死亡离场时，数据统计模块依然能计算出该技能的伤害贡献。新框架的 Tick 函数中调用 `apply_environment_damage(...)` 时必须继续传递。
- **关卡静态地形的空源处理（代码自动注入机制）**：由于我们打算把关卡预设的永久地形也合并进本模块，这类自然地形在配置时无需标注来源参数，它们的原始 `source` 本身就为 `null`。因此在底层的 `TerrainManager.add_static_terrain` 等接口中，**代码内部应自动为其构建并绑定一套统一的默认环境归属**（例如强行注入 `source_name: "Environment", source_id: 0` 置于其内部状态的 `source_fallback` 中）。这样伤害统计模块既能把“关卡环境伤害”独立列出，也保持了地块 JSON 配置的极其纯粹，无需掺杂数据统计的特殊字段。

### 2. 关卡编辑器 / `stage_bridge.gd` 改造

- M5 关卡配置 (`stages_batch.json`) 中的障碍物写法可升级为配置地形。
- 以前是 `"obstacles": [ [1,2], [2,3] ]`。
- 修改为：
  ```json
  "terrains": [
      {
          "terrain_id": "terrain_rock",
          "cells": [[1,2], [2,3]]
      },
      {
          "terrain_id": "terrain_poison",
          "cells": [[4,5], [4,6]]
      }
  ]
  ```
- `StageBridge` 或 `Battlefield` 解析时，直接调用 `TerrainManager.add_static_terrain`。
- 这样，静态毒沼和石头障碍可以直接享受与临时技能同源的地形系统渲染及 Tick 数据，完美统一。

### 3. `hex_grid.gd` 清理

- `_obstacle_cells` 与 `_terrain_cells` 的概念在底层合并为单纯颜色的叠加。
- `TerrainManager` 成为管理场地格子渲染颜色的唯一真理来源（Source of Truth）。

## 四、验证计划

1. **临时特效测试**：施放一个 M3/M5 配置中的“烈火区”功法，验证是否正确读取 JSON 颜色并每秒平滑跳伤害。
2. **永久关卡地形测试**：在关卡 JSON 埋入 `terrain_rock` 和 `terrain_poison` 永久格。开局后，验证是否自动生成阻挡，且站在永久毒区上的角色是否能持续受到伤害/Debuff 影响。
3. **Mod 验证**：在 `data/terrains/` 添加新文件 `terrain_lava.json`，无需改动任何 GDScript 代码即可使用 `op: "create_terrain", "terrain_type": "lava"` 创建新机制。
