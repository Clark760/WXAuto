# GDScript可读性与编码规范

> 本规范用于约束生产代码的可读性、中文注释、模块边界与 Godot 风格一致性。它不是建议，而是架构治理规则的一部分。

## 1. 目标

本规范解决四类问题：

1. 禁止为了压低行数而压缩代码排版。
2. 保证代码对人类读者友好，而不是只对解释器友好。
3. 保证关键业务逻辑有足够的中文注释承载设计意图。
4. 保证 GDScript 写法尽量贴近 Godot 官方推荐风格。

## 2. 基本原则

1. 优先可读，再谈压缩。
2. 注释解释“为什么”和“边界”，不要复述语法。
3. 模块拆分优先于格式压缩。
4. Facade 只做兼容和装配，不做业务堆叠。
5. `20%` 中文注释和 `8%` 空行的目标，是提升函数、变量和状态流转的人类可读性，不是配额游戏。

## 3. Godot 风格要求

1. 使用 `snake_case` 命名函数、变量、文件名。
2. 常量使用 `UPPER_SNAKE_CASE`。
3. 能明确类型时必须写类型标注。
4. 函数之间保留空行，逻辑块之间保留空行。
5. 禁止把多段业务逻辑硬塞到同一个超长函数中。
6. 禁止用动态 `call()` 代替明确接口，除非处于兼容边界。

## 4. 中文注释要求

1. 作用范围：`scripts/**/*.gd`，排除 `scripts/tests/**` 和 `tools/**`。
2. 对“改动文件”强制执行。
3. 文件级口径：
   - 中文注释行 / 非空代码行 `>= 20%`
4. 统计定义：
   - 中文注释行：包含 `#` 且注释内容中含中文字符的行。
   - 非空代码行：去掉空行与纯注释行后的代码行。
5. 代码行数少于 `20` 的小文件可以不做比例硬性要求，但关键函数仍应写中文注释说明意图。

注释内容必须优先覆盖：

1. 兼容层存在原因。
2. 复杂分支的判定依据。
3. 生命周期或状态同步边界。
4. 与旧接口保持一致的限制条件。

### 4.1 注释分布规则

1. 中文注释必须优先写在函数入口、关键分支和重要参数旁边，不能主要堆在文件头部。
2. 生产代码中的函数，默认都应有紧邻函数定义的中文注释；至少说明职责、输入口径或返回语义中的一项。
3. 多参数函数如果包含 `effect`、`context`、`summary`、`runtime_gateway` 这类关键参数，函数注释必须说明这些参数的职责边界，避免调用方误用。
4. 注释率默认应由“函数入口注释 + 关键参数边界注释 + 复杂分支注释”共同承担，不能主要依赖文件头说明。
5. 允许保留简短文件头说明，但只用于交代模块定位，不能代替函数级注释。
6. 重要参数说明必须贴近实际函数，而不是集中写在 `文件说明` 或 `维护约束` 里统一凑字数。
7. 对生产函数，若参数里出现 `effect`、`context`、`summary`、`runtime_gateway`、`source`、`target` 这类运行时关键对象，注释中至少应明确其中一个参数的职责边界。
8. `# 文件说明：`、`# 维护约束：` 这类命名注释块每块不得超过 `6` 行，且不能作为凑够中文注释比例的主要来源。
9. 禁止为了满足注释率，堆砌重复表述、同义反复或与函数实现无直接对应关系的车轱辘话。
10. 如果注释删掉后，读者仍然不知道函数为什么存在、参数该怎么传、边界在哪里，说明注释写错位置了。
11. 禁止把中文注释主要堆在文件尾的 `附录`、`边界附录`、`职责附录` 一类长注释块里充数。
12. 注释率应优先由函数入口注释、关键状态字段注释、复杂分支注释和节点绑定分组注释承担。
13. 如果一个注释块连续重复“只负责/不负责”但不能帮助读者理解当前函数或当前代码块，视为无效注释。
14. 对状态密集文件，优先解释关键变量和状态同步点，不要把说明集中到文件末尾统一宣言。

## 5. 可读性硬约束

1. 单行长度不得超过 `120` 字符。
2. 函数体不得超过 `80` 行。
3. 大文件（`>= 200` 行）空行占比不得低于 `8%`。
4. 禁止单行函数体：
   - 禁止 `func foo(): return bar`
5. 禁止单行控制流：
   - 禁止 `if cond: return`
   - 禁止 `for item in items: do_work()`
   - 禁止 `while cond: step()`
   - 禁止 `match value: handle()`
6. 禁止通过以下方式规避行数限制：
   - 删除必要空行
   - 删除说明性注释
   - 把多条语句并到同一行
   - 把多个职责继续塞进同一文件但仅靠排版压缩过线
   - 用超长 `文件说明` / `维护约束` 注释块代替函数级注释
   - 用文件尾大段 `附录` / `边界附录` 代替函数级、变量级、分支级说明
   - 在文件尾或无关位置集中补空行，只为了把空行占比凑到 `8%`

## 6. 模块拆分要求

1. 文件超过 `600` 行时，必须按职责评审是否拆分。
2. 文件超过 `900` 行时，禁止继续在原文件上叠加实现。
3. 任何 `> 900` 行文件只要被修改，必须在同一轮改动中同步完成模块迁移，不允许只做排版压缩。
4. Facade 文件只允许保留：
   - 对外公开 API
   - 装配逻辑
   - 兼容网关
5. Facade 文件不得继续承载：
   - `op match` 分发实现
   - 大段纯业务逻辑
   - 复杂状态机分支

推荐拆分方向：

1. 查询逻辑独立成 query/service。
2. 副作用执行独立成 ops/handlers。
3. 汇总与事件整理独立成 emitter/collector。
4. 兼容入口与真实实现分离。

## 7. 反例与正例

反例：

```gdscript
func execute(): return _dispatcher.call(self)
if target != null: return _apply_damage(target)
for unit in units: _tick_unit(unit)
```

正例：

```gdscript
## 这里说明 facade 为什么仍然保留在这一层。
func execute() -> void:
	# `context` 只承载运行时依赖，不能在这里继续追加业务判定。
	_dispatcher.execute(self)

if target != null:
	# 命中目标后立即结算伤害，避免后续逻辑继续污染当前分支。
	return _apply_damage(target)

for unit in units:
	_tick_unit(unit)
```

## 8. 执行与豁免

1. 本规范由 `tools/architecture_guard.ps1` 执行。
2. 豁免只能写入 `tools/baselines/architecture_guard_exceptions.json`。
3. 禁止在源码里通过 suppression comment 绕过门禁。
4. 任何临时豁免都必须写明原因和移除阶段。

## 9. 测试数据要求

1. 当测试依赖的数据在生产 Mod 中不存在时，必须优先在 `mods/test/` 下创建测试专用 Mod。
2. 禁止把成组测试数据长期内联在测试脚本里充当数据源。
3. 禁止继续引用仓库中不存在的历史路径来伪装“有测试覆盖”。
4. 测试专用 Mod 应只包含该测试最小必需的数据文件，并保持目录结构与正式 Mod 一致。
5. 如果测试为了临时回归补了 `mods/test/` 数据，相关测试和说明文档必须同时更新到新路径。
